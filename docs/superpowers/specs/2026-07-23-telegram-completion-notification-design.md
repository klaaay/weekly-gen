# Telegram 周刊完成通知设计

## 背景

`weekly-gen` 可以单独执行某个周刊，也可以通过 `src.main` 并发执行多个周刊。每个周刊最终都会调用 `extract_links_and_summarize`，依次抓取文章、生成摘要并写入 Markdown 文件。

用户希望某个周刊处理完该期的全部文章链接后，立即通过 Telegram 机器人收到一条消息。不同周刊之间无需互相等待。

## 目标

- 每个周刊完成该期全部文章链接的处理和写入后，立即发送一条 Telegram 消息。
- 通知包含周刊名称、处理链接数、摘要路径和完成时间。
- Telegram 发送失败不改变周刊的执行结果，也不阻塞其他周刊。
- token 和 chat ID 只保存在本地环境变量中，不进入 Git 或日志。
- 直接运行单个周刊脚本和通过 `src.main` 运行时，行为一致。

## 非目标

- 不发送抓取开始、抓取失败或全批次汇总通知。
- 不为 Telegram 发送失败自动重试。
- 不发送摘要文件内容或附件。
- 不改变现有文章抓取、摘要生成和去重逻辑。

## 方案选择

采用公共链接处理函数的完成回调。

不在 `src.main.run_scraper` 返回后直接发送。部分爬虫在重复期刊、页面异常或没有链接时会提前返回，而主入口仍会把正常返回视为成功。这会产生误报。

不在每个周刊脚本中分别实现 Telegram 请求。重复实现会增加维护成本，也容易在新增周刊时遗漏。

## 组件设计

### `src/utils/telegram_notifier.py`

该模块负责 Telegram 通知：

- 从 `TELEGRAM_BOT_TOKEN` 和 `TELEGRAM_CHAT_ID` 读取配置。
- 创建绑定周刊显示名称的完成回调。
- 将完成事件格式化为纯文本。
- 通过 Telegram Bot API 的 `sendMessage` 方法发送消息。
- 捕获配置缺失、超时、网络异常、HTTP 错误和 Telegram API 错误。
- 返回发送结果，但不向抓取流程抛出异常。

请求使用 HTTPS `POST`，超时为 10 秒。请求数据放在 JSON body 中。日志不得输出 token，也不得输出包含 token 的完整 URL。

### `src/utils/extract_links_and_summarize.py`

为 `extract_links_and_summarize` 增加可选完成回调。公共模块定义 `ExtractionCompletion` 数据对象，回调接收该对象。数据对象包含：

- `processed_count`：实际完成循环处理的链接数。
- `summary_file`：函数计算出的最终摘要文件路径。
- `completed_at`：Asia/Shanghai 时区的完成时间。

回调必须在链接循环结束且每篇文章已完成文件写入后执行。只有 `processed_count > 0` 时才触发。

公共函数对回调再加一层异常保护。即使回调实现意外抛出异常，也只能记录警告，不能改变函数返回值或周刊状态。

### 周刊脚本

每个周刊脚本通过通知模块创建带有自身显示名称的 Telegram 完成回调，并传给 `extract_links_and_summarize`。周刊名称绑定在回调中，不属于公共链接处理函数的数据。

这项接线不改变抓取顺序、链接筛选、摘要生成、文件命名或状态持久化。

## 触发语义

满足以下条件时发送：

1. 周刊前置请求成功。
2. 找到至少一个待处理的有效文章链接。
3. 所有链接均已依次调用内容获取函数。
4. 每篇结果均已追加到摘要文件。
5. 链接循环正常结束。

以下情况不发送：

- 去重逻辑判定该期已处理。
- 前置页面请求失败。
- 没有找到有效文章链接。
- 链接循环中出现未处理异常，导致循环提前终止。

文章内容获取函数可能把单篇失败转换为“无法总结”结果。此时该链接仍属于“已处理”，会计入 `processed_count`。完成通知表示全部链接已处理，不表示每篇文章都成功生成有效摘要。

## 消息格式

消息使用纯文本，不启用 `parse_mode`：

```text
✅ 周刊抓取完成

周刊：Frontend Focus
已处理：18 个链接
摘要：outputs/2026-07-23_frontendfoc_..._summary.md
时间：2026-07-23 15:30:12
```

摘要路径使用项目相对路径。若无法得到摘要路径，则显示 `未生成摘要文件`。

## 配置

本地 `.env` 保存真实配置：

```env
TELEGRAM_BOT_TOKEN=<bot-token>
TELEGRAM_CHAT_ID=<chat-id>
```

`.env` 已被 `.gitignore` 忽略。README 和其他受 Git 管理的文件只记录变量名和占位符。

缺少任一环境变量时，通知模块输出一条不含敏感信息的跳过提示，并返回失败状态。

## 错误处理

- 配置缺失：跳过发送，记录警告。
- 请求超时或网络异常：记录简短错误，返回失败。
- HTTP 非成功状态：记录状态码，不记录请求 URL。
- 响应不是合法 JSON：记录响应格式错误。
- Telegram 返回 `ok: false`：记录不含凭据的 API 描述。
- 完成回调抛出异常：公共链接处理函数捕获并记录，继续返回文章结果。

发送失败不重试。Telegram 可能已接收请求但客户端没有收到响应，自动重试会造成重复消息。

## 测试策略

### 通知模块

- 配置完整时向 `sendMessage` 发送正确的 `chat_id` 和 `text`。
- 消息包含周刊名称、处理链接数、摘要路径和完成时间。
- 缺少 token 或 chat ID 时不发起 HTTP 请求。
- 网络异常、HTTP 错误、无效 JSON 和 `ok: false` 均返回失败且不抛出。
- 日志和异常文本不包含 token。

### 完成回调

- 所有链接处理并写入后触发一次。
- 回调接收到正确的处理数量和最终摘要路径。
- 没有链接时不触发。
- 链接循环异常中断时不触发。
- 回调抛出异常时，函数仍返回已处理文章数据。

### 回归验证

- 运行 Telegram 通知相关测试。
- 运行完整 `pytest` 测试套件。
- 检查 Git diff，确认没有 token 或 chat ID 进入受 Git 管理的文件。

## 验收标准

- 一个周刊处理完该期全部有效链接后，10 秒内尝试发送一条完成消息。
- 并发执行多个周刊时，每个完成的周刊各发送一条消息。
- 通知失败不会使周刊失败，也不会影响其他周刊。
- 重复期刊、前置失败或没有链接时不发送完成消息。
- Git diff 和测试日志不包含真实 token。
