### [两种错误](https://evanhahn.com/the-two-kinds-of-error/)

**原文标题**: [The two kinds of error](https://evanhahn.com/the-two-kinds-of-error/)

软件错误可分为预期错误与意外错误两类，前者是正常操作中可预见且应妥善处理的状况，后者则通常由程序缺陷引起且可能导致系统崩溃。

- 🧑‍💻 预期错误属于正常操作范畴（如用户输入无效数据、网络中断），并非开发者过错，应通过返回错误结果等方式进行恢复处理，并采用 WARN/INFO 级别日志记录
- 🐛 意外错误意味着程序存在缺陷（如空指针异常、断言失败），属于开发者责任，通常应允许程序崩溃并记录 ERROR/FATAL 级别日志以暴露问题
- ⚖️ 错误分类界限取决于应用场景：原型脚本可忽略大部分错误，关键系统（如太空探测器）则需预期几乎所有异常情况
- 🔧 不同编程语言对错误的处理哲学各异（如 Rust 强制处理预期错误，JavaScript 则较宽松），开发者应根据项目类型选择合适工具
- 📈 提升软件可靠性的趋势是不断扩大预期错误范围，主动处理更多潜在异常情况

### [对抗基于查询的编译器](https://matklad.github.io/2026/02/25/against-query-based-compilers.html)

**原文标题**: [Against Query Based Compilers](https://matklad.github.io/2026/02/25/against-query-based-compilers.html)

基于查询的编译器是当前热门技术，它通过将编译过程建模为函数调用图来实现增量计算，从而在输入变化时仅重新计算受影响的部分，并支持“早期截止”优化。然而，这种方法的效率受限于源语言的依赖结构：若语言特性导致细粒度依赖（如 Rust 的宏和特质系统），增量更新可能涉及大量计算；反之，若语言设计允许粗粒度独立处理（如 Zig 的文件级解析），则可减少查询需求，提升性能。因此，理想方案是结合语言语义手动划分编译任务，并优先采用直接更新等非查询方法，以简化实现并提高速度。

- 🧩 基于查询的编译器通过函数调用图实现增量计算，仅重新计算受输入变化影响的部分，并支持“早期截止”优化。
- ⚡ 增量编译的核心目标是实现 O(1) 的响应时间，使更新工作量与代码变更规模成正比，而非整个代码库的大小。
- 🧱 该方法的有效性受语言依赖结构限制：若变更引发“雪崩效应”（如加密函数），增量更新可能无法高效实现。
- 🔍 潜在依赖问题可能导致增量引擎消耗资源确认无变更，即使实际影响较小。
- 📚 语言设计至关重要：Zig 通过允许文件独立解析和名称解析，减少了查询需求；而 Rust 的宏和特质系统则强制细粒度依赖跟踪。
- 🛠️ 替代方案包括手动划分编译阶段（如先并行处理文件摘要，再顺序解析签名），并优先采用非查询方法（如直接映射更新）以提升性能。
- 🔗 类似方法可应用于链接等场景，例如通过就地修补而非完全重建来实现增量更新。

### [什么是 OAuth？](https://leaflet.pub/p/did:plc:3vdrgzr2zybocs45yfhcr6ur/3mfd2oxx5v22b)

**原文标题**: [What is OAuth?](https://leaflet.pub/p/did:plc:3vdrgzr2zybocs45yfhcr6ur/3mfd2oxx5v22b)

OAuth 是一种用于委托授权的开放标准，其核心设计旨在安全地允许第三方应用在用户授权下访问其资源，而无需分享密码。

- 🔐 OAuth 的核心是委托授权：通过标准化流程，让用户可安全授权第三方应用访问其资源，无需直接提供密码。
- 📜 历史背景源于需求：2006 年左右，Twitter 等 Web 2.0 网站需要支持第三方客户端登录，但当时方案如 Flickr 或 AWS 的方法都不安全且不统一，推动了 OAuth 标准的建立。
- 🧩 OAuth 与 OpenID Connect (OIDC) 关联：OIDC 基于 OAuth 实现身份验证，类似于“魔法链接”机制，通过发送秘密到用户专属位置来验证身份。
- ⚙️ 设计本质简单：OAuth 前半部分处理用户同意下发送可重复使用的秘密给委托方，后半部分定义委托方如何使用该秘密代表用户发起请求。
- 🌐 标准演变为框架：OAuth 相关规范更像一个框架，允许灵活实现，类似 HTML，不同浏览器无需支持所有功能。
- 💡 实施关键在理解目的：使用 OAuth 时应先明确“做什么”和“为什么”，再处理“如何做”的机械细节，避免因复杂机制而忽略原始目标。
