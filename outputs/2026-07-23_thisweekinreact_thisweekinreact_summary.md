### [我们还需要状态管理库吗？ — Neciu Dan](https://neciudan.dev/do-we-need-state-management-libraries)

**原文标题**: [Do we need state management libraries anymore? — Neciu Dan](https://neciudan.dev/do-we-need-state-management-libraries)

本篇文章探讨了在现代 React 应用中是否还需要状态管理库，通过深入分析 Redux、Zustand、Jotai、MobX 和 Valtio 等主流库的内部实现，并亲手构建了一个简化版 Zustand，最终得出结论：对于大多数应用，结合 React 内置功能和少量自定义代码已足够，但复杂客户端状态场景下，成熟库仍有其价值。

- 📜 **Redux 核心是发布/订阅模式**：通过 reducer 纯函数管理状态，action 对象可序列化，支持时间旅行调试，但旧版样板代码多，现代 Redux Toolkit 简化了流程。
- 🐻 **Zustand 简化了 Redux 机制**：去除了 action 和 reducer 约定，支持直接更新状态，体积小（约 1KB gzipped），通过 `useSyncExternalStore` 连接 React。

- 🔄 **MobX 通过 Proxy 实现自动追踪**：将状态变为可观察对象，通过 `autorun` 和 `observer` 自动收集依赖，但需注意解构和日志等陷阱。
- 🪞 **Valtio 结合 Proxy 与不可变快照**：写操作通过 Proxy 捕获，读操作返回冻结快照，利用 `proxy-compare` 实现精确重渲染，但需区分读写对象。
- 🔗 **所有库最终都依赖 `useSyncExternalStore`**：该 React 18 钩子解决了并发渲染下的“撕裂”问题，但外部状态无法享受时间切片和过渡更新。

- 🧩 **第二层：复合组件**：当组件 props 膨胀时，通过子组件暴露插槽（如 Item.Root、Item.Content），让调用者自由组合布局，内部通过轻量 context 协调。

- 🏷️ **命名规范**：普通名称表示 props 组件，Composer 后缀表示消费契约，Provider 后缀表示数据源，让代码意图一目了然。
- ⚠️ **何时升级**：仅当遇到具体痛点时升级，如 props 膨胀、多数据源、状态跨边界，避免为统一性或单一数据源而升级。
- 💡 **核心原则**：默认使用最简方案，每个层级可逆但实践中难以逆转，因此从低层起步。

- 🔄 **渐进式采用策略**：调试模式下双编码校验确保正确性；未迁移组件回退到原有 map 路径；组件作者无需修改代码。

- 💡 **实践建议**：根据数据形状选择工具。高频数据应通过轻量级通道处理，避免通过重型库。低端 Android 设备上，完整服务端状态客户端在写入频率适中时仍可接受。

- 🚀 **核心创新**：用整数索引替代指针，将 AST 节点存储在连续数组中，消除指针追踪和缓存未命中

**原文标题**: [Fetch Is Not Enough | James M Snell](https://www.jasnell.me/posts/fetch-is-not-enough)

### 概述总结
本文探讨了 Fetch API 在服务端运行时中的局限性，指出现有 HTTP API 在协议版本支持、流式传输、隧道协议等方面存在碎片化问题，并提出需要扩展 API 以覆盖 HTTP 的完整能力。

- 🔍 **Fetch 的局限性**：Fetch API 专为浏览器客户端设计，无法表达服务端所需的尾随头、信息响应、全双工流、类型化流重置、优先级、隧道及不可靠数据报等特性。
- 🧩 **运行时碎片化**：Node.js 有四个独立的 HTTP 模块（http、https、http2、quic），Deno、Bun、Workers 虽采用 Request/Response 类型，但 WebSocket、生命周期等实现各异，导致代码不可移植。
- 🔄 **协议扩展需求**：HTTP 已超越简单请求/响应模型，扩展 CONNECT（WebSocket、WebTransport、MASQUE）、胶囊协议、HTTP 数据报等标准化功能缺乏统一 API。
- 🛠️ **未来 API 设计方向**：提出双处理器模型（fetch 处理请求/响应，connect 处理隧道），扩展 Request/Response 支持尾随头、信息响应、优先级，并抽象隧道原始数据流、胶囊协议、数据报三层通信。
- 🌍 **适用场景**：主要面向服务端、边缘和代理运行时，浏览器客户端需求相对有限；多数简单应用仍可继续使用现有 Fetch API。
