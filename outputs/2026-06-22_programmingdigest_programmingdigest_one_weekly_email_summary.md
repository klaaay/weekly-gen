### [停止使用常规提交 - Sumner Evans](https://sumnerevans.com/posts/software-engineering/stop-using-conventional-commits/)

**原文标题**: [Stop Using Conventional Commits - Sumner Evans](https://sumnerevans.com/posts/software-engineering/stop-using-conventional-commits/)

本文批评了 Conventional Commits 标准，认为它优先关注错误的方面（提交类型），而忽略了更重要的范围（scope），且其承诺的优势（如自动生成变更日志和版本号）在实际中难以实现。作者建议采用范围前缀的提交消息格式，并推广了 scopedcommits.com 网站。

- 📝 **范围比类型更重要**：开发人员、调试人员和事故响应者更关心提交的范围（如修改的代码区域），而非类型（如修复或功能）。Conventional Commits 将范围设为可选，优先显示类型，这完全颠倒了优先级。
- ❌ **类型冗余且限制性**：提交描述通常已暗示类型（如“修复 bug”），额外添加类型浪费空间，且可能不准确（如一个提交同时包含修复、重构和新功能）。
- 🚫 **自动生成变更日志不切实际**：变更日志面向用户，关注功能变化；提交日志面向开发者，关注代码演变。两者目标不同，强行合并导致结果不佳（如多个提交实现一个功能，或回滚提交造成混乱）。
- ⚠️ **自动版本号提升不可靠**：回滚、意外破坏或组合修复等情况会导致工具错误计算版本号，而重写历史会破坏提交日志的可靠性。
- 🔒 **触发构建流程是坏主意**：仅基于提交类型触发安全检查可能被绕过（如伪装为文档提交的恶意代码），应使用 `git diff` 检查实际修改的文件。
- 🛠️ **难以应用于实际项目**：定义自定义类型困难，且常与公司要求（如添加工单号）冲突，导致范围字段被滥用。
- 🌟 **更好的替代方案**：成功项目（如 Linux、Go、NixOS）使用范围前缀的提交消息，例如“subsystem: description”或“package: description”，这更自然且实用。
- 💡 **推广 scopedcommits.com**：作者创建网站倡导范围优先的提交格式，并区分变更日志和提交日志的管理。

### [停止使用 JWT · GitHub](https://gist.github.com/samsch/0d1f3d3b4745d778f78b230cf6061452)

**原文标题**: [Stop using JWTs · GitHub](https://gist.github.com/samsch/0d1f3d3b4745d778f78b230cf6061452)

本文强烈建议停止使用 JWT 进行用户会话管理，并推荐使用传统的 cookie 会话机制。

- 🚫 **JWT 不适合会话管理**：JWT 规范专为极短生命周期的令牌（约 5 分钟）设计，而会话需要更长的有效期。
- 🔒 **无状态认证不安全**：真正安全的认证必须依赖状态（如数据库），既然需要数据存储，不如直接使用完整的会话机制。
- 💡 **会话更简单高效**：使用 JWT 存储简单会话令牌既低效又缺乏灵活性，而传统 cookie 会话更直接且无额外优势。
- ⚠️ **JWT 规范存在安全缺陷**：安全专家不信任 JWT 规范，其原始设计甚至允许伪造令牌，建议使用更安全的替代方案（如 PASETO）。
- 🏢 **Google 的 JWT 用法被误解**：Google 仅将 JWT 用于单点登录传输，而非浏览器中的用户会话，其实现有安全专家维护。
- 📉 **“无状态更好”是误解**：真正安全的无状态认证需要巨大资源，且“无状态”本身是一个谎言。
- 🛠️ **会话设置并不复杂**：大多数 Web 框架内置会话支持（如 Express 的 express-session 中间件），文档即可指导完成配置。
- 📱 **移动端同样适用会话**：后端实现与 Web 无异，只需确保客户端正确发送 cookie 即可。
- 🔄 **JWT 的真正用途**：JWT 适用于极短生命周期的授权令牌（如微服务间验证），而非长期会话管理。
