**原文标题**: [Two React Design Choices Developers Don’t Like—But Can’t Avoid - DEV Community](https://dev.to/playfulprogramming/two-react-design-choices-developers-dont-like-but-cant-avoid-d6g)

本文探讨了 React 中两个常被开发者诟病的设计选择——延迟状态提交和 Effect 依赖数组，并指出这些并非 React 的任意决策，而是异步 UI 系统中不可避免的约束。作者通过 Solid 框架的开发经验，说明异步操作如何迫使任何前端框架都必须隔离提交与拆分副作用，以保持 UI 的一致性。这并非模仿 React，而是承认异步环境下的基本物理规律。

- 🌀 **异步操作迫使状态提交隔离**：在异步世界中，若衍生值未就绪，框架必须延迟状态提交，否则会导致 UI 与数据不一致，破坏确定性。

- 🔧 **编译器无法根本解决问题**：编译器虽能提取依赖，但无法改变异步的运行时语义，无法保证提交隔离与副作用拆分的正确性。
- 🚀 **框架设计的成熟方向**：接纳异步约束并非妥协，而是构建更健壮系统的必然选择，使异步成为架构的一等公民，而非边缘情况。

- 🛠️ **优化三：构建时隔离的服务器快速路径**：在保证正确性的通用算法基础上，为服务器端常见情况添加专用快速路径，并利用构建工具的死代码消除功能，确保这些路径不会增加客户端包体积。
- 🗑️ **优化四：避免使用 `delete` 操作**：在性能敏感的热路径中，用将属性设为 `undefined` 或创建新对象的方式来替代 `delete` 操作，以避免改变对象形状导致引擎优化失效。
- 📊 **独立验证与显著成果**：第三方基准测试证实，优化后服务器在高压负载下成功率从 75.52% 提升至 100%，吞吐量翻倍，尾部延迟降低数百倍，事件循环利用率显著改善，为应对流量高峰提供了更大余量。

### [理解 React 中的'key'属性 | React 内部解析](https://inside-react.vercel.app/blog/making-sense-of-key-prop-in-react)

**原文标题**: [Making sense of 'key' prop in react | Inside React](https://inside-react.vercel.app/blog/making-sense-of-key-prop-in-react)

React 的 `key` 属性用于在列表渲染中为每个元素提供稳定且唯一的标识，帮助 React 在更新时准确识别哪些元素被移动、添加或删除，从而避免不必要的 DOM 操作和状态错乱问题。

- 🔑 **`key` 的作用**：为列表中的每个元素提供唯一标识，使 React 能正确追踪元素变化，优化渲染性能。
- 🧩 **无 `key` 的问题**：React 依赖元素位置进行对比，可能导致状态错乱（如排序后按钮状态跟随错误元素）或性能下降。
- ⚠️ **索引作为 `key` 的局限**：使用数组索引作为 `key` 仅在列表静态且无内部状态时有效，否则可能引发状态错乱。
- 🚫 **随机值作为 `key` 的危害**：如 `Math.random()` 会导致每次渲染都重新创建所有元素，破坏状态保持。
- ✅ **理想 `key` 的选择**：应使用稳定且唯一的标识（如数据 ID），确保在重新渲染时保持不变。
- 🔄 **`key` 对组件的影响**：为组件添加 `key` 会赋予其新身份，导致组件重新挂载（状态重置）；非列表场景通常无需 `key`。
- 📊 **算法效率**：`key` 不改变 React 对比算法的 O(N) 复杂度，但能减少错误的 DOM 更新。
