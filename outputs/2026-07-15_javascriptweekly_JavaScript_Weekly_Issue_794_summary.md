Bun 运行时从 Zig 语言重写为 Rust 语言，通过 AI 辅助在 11 天内完成，解决了大量内存安全漏洞，提升了稳定性和性能。

- 🚀 **项目起源**：Bun 最初是作者在 2021 年用 Zig 语言从零开始构建的，因其对性能的极致追求和底层控制能力而选择 Zig。
- 🎯 **庞大范围**：Bun 集成了 JS/TS/CSS 转译器、包管理器、测试运行器、HTTP 服务器等大量功能，是一个野心勃勃的全栈工具。
- 🐛 **稳定挑战**：Zig 的手动内存管理导致大量 use-after-free、double-free 和内存泄漏等严重 bug，仅一个版本就修复了 15 个以上此类问题。
- 🔄 **重写决策**：团队评估后认为，用 Rust 重写可通过其所有权系统和 `Drop` trait 从编译器层面杜绝此类内存错误，而非依赖代码审查。
- 🤖 **AI 驱动重写**：作者使用 Claude Fable 5 和 Claude Code 的动态工作流，在 11 天内完成了 535,496 行 Zig 代码到 Rust 的机械翻译。

- ✅ **质量保障**：通过语言无关的 TypeScript 测试套件（百万级断言）、对抗性代码审查和持续集成，确保重写后功能完全一致。
- 🐛 **审查发现**：对抗性审查在合并前发现了 3 个关键 bug，包括异步 close 导致的 use-after-free、负时间戳处理错误和 `unwrap_or` 的急切求值问题。
- 📉 **性能提升**：Rust 版本在 HTTP 吞吐量上提升 2.8%-4.8%，应用构建速度提升 2.2%-4.7%，二进制体积缩小约 20%。
- 🧠 **内存优化**：通过 `Drop` trait 自动清理，修复了所有可检测的内存泄漏，例如 `Bun.build()` 重复调用时的内存泄漏从 6.7GB 降至 609MB。
- 🔧 **回归控制**：重写引入了 19 个已知回归，均已被修复，主要源于 Zig 和 Rust 在语义上的细微差异（如 `debug_assert!` 宏的副作用问题）。
- 💰 **成本效益**：整个项目消耗约 59 亿输入 tokens 和 6.9 亿输出 tokens，API 成本约 16.5 万美元，但相比人工团队一年工期节省了大量资源。

### [GitHub - tc39/proposal-await-dictionary: 向 ECMAScript 添加 Promise.allKeyed 的提案 · GitHub](https://github.com/tc39/proposal-await-dictionary)

**原文标题**: [GitHub - tc39/proposal-await-dictionary: A proposal to add Promise.allKeyed to ECMAScript · GitHub](https://github.com/tc39/proposal-await-dictionary)

该提案旨在为 JavaScript 添加`Promise.allKeyed`和`Promise.allSettledKeyed`方法，以解决并行执行多个 Promise 时，使用`await`逐个等待导致的水瀑布效应，以及`Promise.all`基于顺序而非名称可能引发的混淆问题。

- 📜 **核心动机**：解决`await`逐个执行 Promise 造成的性能瓶颈，以及`Promise.all`按顺序返回结果可能导致的变量混淆。
- 🔑 **解决方案**：引入`Promise.allKeyed`，允许通过对象键名并行执行 Promise，并保持结果与键名的对应关系，避免变量污染和未处理的 Promise 拒绝风险。
- 🛠️ **附加 API**：同时提供`Promise.allSettledKeyed`，用于处理所有 Promise 完成后的结果，包括成功和失败状态。
- 🤔 **设计考量**：仅处理自有可枚举属性（包括符号键），不进行深拷贝，以保持与现有 JavaScript 内置 API 的一致性。

- 🚀 **解决痛点**：当前需用 `get()` 和 `delete()` 两次查找同一键，存在性能开销、易写错（如先删后读）和重复实现的问题。

- 💡 **命名考量**：`take` 简洁且与 Rust 一致；`getAndDelete` 是备选，能明确体现"先读后删"的关系。
- 🛠️ **Polyfill 可用**：提供兼容性 polyfill，通过 `get` 和 `delete` 方法实现，可安全安装到 `Map` 和 `WeakMap` 原型上。

### [介绍 Precursor：通过持续客户端信号检测代理行为](https://blog.cloudflare.com/introducing-precursor/)

**原文标题**: [Introducing Precursor: detecting agentic behavior with continuous client-side signals](https://blog.cloudflare.com/introducing-precursor/)

### 概述总结  
Cloudflare 推出 Precursor，一种基于客户端会话的验证系统，通过持续收集用户行为信号（如鼠标移动、键盘节奏等）来区分人类与自动化流量，弥补了传统验证码仅覆盖关键节点的不足。该系统以隐私设计为原则，不记录具体按键内容，仅分析行为模式，并实时整合到 Cloudflare 的机器人防护中。  

- 🛡️ **Precursor 核心功能**：通过动态注入 JavaScript，持续采集用户交互行为信号（如鼠标轨迹、键盘节奏、页面可见性），实时分析会话级行为模式，区分人类与自动化流量。  
- 🔍 **解决可见性缺口**：传统验证码（如 Turnstile）仅保护登录、结账等关键节点，而 Precursor 覆盖完整用户旅程，捕捉请求之间的行为数据。  
- 🧠 **行为信号示例**：人类鼠标移动受生理限制（如手腕旋转、手抖），而机器人常呈现直线轨迹、固定速度或完美贝塞尔曲线；键盘信号仅记录节奏而非具体按键。  
- ⚙️ **工作原理**：轻量脚本注入页面 → 采集事件并缓冲 → 边缘服务器评估 → 跨会话累积数据 → 影响机器人评分（如降低人类用户干扰）。  
- 🔒 **隐私设计**：仅收集行为模式（如鼠标轨迹、按键间隔），不记录具体按键内容或用户身份，数据不暴露给客户仪表盘。  
- 📊 **会话分析功能**：Security Analytics 新增会话级视图，可追踪典型用户行为、识别异常会话（如自动化迹象）。  
- 🚀 **未来扩展**：持续增加行为信号类型，优化会话级防护策略，并推出可视化工具。  
- 💡 **启用方式**：免费开放至 GA 发布，可在 Cloudflare 仪表盘直接开启，支持低摩擦观察模式或强制验证模式，无需修改应用代码。
