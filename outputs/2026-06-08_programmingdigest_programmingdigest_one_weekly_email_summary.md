### [应该将 RGB 值归一化为 255 还是 256？](https://30fps.net/pages/255-vs-256-division/)

**原文标题**: [Should you normalize RGB values by 255 or 256?](https://30fps.net/pages/255-vs-256-division/)

本文讨论了在图像处理中将 8 位 RGB 整数转换为浮点数时，除以 255 还是 256 的争议。标准方法（除以 255）将 0 映射到 0.0、255 映射到 1.0，便于检测黑色等极端值，但存在极值区间宽度减半和浮点值不精确的问题。替代方法（加 0.5 后除以 256）将值置于整数中间，理论上更精确且分布均匀，但会失去零映射到零的便利性。结论是：处理他人图像时推荐标准方法（除以 255）；若控制整个流程且不依赖零映射，可考虑替代方法（除以 256）。

- 📊 **标准方法（除以 255）**：将整数 0-255 映射到 0.0-1.0，便于检测黑色，但极值区间（0 和 255）宽度减半，导致输出时这些值出现频率较低。
- 🔄 **替代方法（加 0.5 后除以 256）**：将整数映射到浮点数的中间位置，分布更均匀，但无法直接检测黑色，且与标准方法不兼容。
- ⚖️ **量化误差对比**：替代方法理论上更精确（平均绝对误差 1/1024 vs 1/1020），但实际图像处理中差异极小，且若图像按标准方法保存，用替代方法加载会引入额外误差。
- 🧩 **量化器类型**：标准方法类似“中升式”（mid-riser），替代方法类似“中平式”（mid-tread），两者在零值附近行为不同。
- 💡 **实际建议**：处理他人图像时，使用标准方法（除以 255）；若控制整个流程且不依赖零映射，可考虑替代方法（除以 256）以提升精度。

### [每个字节都重要 | Farid Zakaria 的博客](https://fzakaria.com/2026/06/01/every-byte-matters)

**原文标题**: [Every byte matters | Farid Zakaria’s Blog](https://fzakaria.com/2026/06/01/every-byte-matters)

### 概述总结
本文探讨了数据结构布局对 CPU 缓存性能的影响，通过对比数组结构体（AoS）与结构体数组（SoA）的访问模式，揭示了缓存行利用率和工作集大小如何显著影响程序执行速度。

- 📊 **缓存层次结构**：L1 缓存访问约 1-2ns，L2 约 4-5ns，L3 约 10-15ns，主存约 60-100ns，缓存行大小为 64 字节
- 🧩 **AoS vs SoA**：结构体数组（SoA）可将相关数据紧凑排列，单次缓存行获取 64 个布尔值，而数组结构体（AoS）每个缓存行仅含 1 个完整对象
- ⚡ **性能差异**：当结构体大小为 1KiB 时，SoA 布局可带来高达 30 倍的性能提升
- 🔄 **顺序访问优势**：CPU 预取器能高效处理顺序访问，使数据几乎无需等待内存读取
- 🎲 **随机访问挑战**：哈希表、树等随机访问模式无法预取，工作集大小直接决定性能层级
- 📏 **工作集影响**：512 个 64B 结构体可在 L1 缓存（~3ns）内处理，而 128B 结构体已溢出至 L2（~11ns）
- 📉 **缓存阶梯效应**：随机访问时，所有结构体大小均呈现阶梯式延迟增长，但较大结构体更早触发性能降级

- 🐾 **宠物定义**：指不可替代、需手动维护的服务器（如邮件服务器、主从数据库），一旦故障需紧急处理。

- 💡 **核心价值**：比喻的关键在于强调服务器的可丢弃性，帮助 IT 人员快速理解新旧架构的根本差异，避免因误用而失去教育意义。
- 🔄 **正确使用建议**：若需改编比喻，应明确说明新语境，并引用原始定义以保持历史准确性。

### [面向开发者的 AI 工程 | 博客 | Luca Cavallin](https://www.lucavallin.com/blog/ai-engineering-for-developers)

**原文标题**: [AI Engineering for Developers | Blog | Luca Cavallin](https://www.lucavallin.com/blog/ai-engineering-for-developers)

### 概述总结
本文为有后端开发经验的工程师提供了一份全面的 AI 工程指南，涵盖从基础概念到生产部署的完整流程，强调构建围绕模型的系统而非模型本身。

### 关键要点
- 🧠 **核心转变**：AI 工程的核心是构建围绕基础模型的系统，而非模型本身；模型输出具有概率性，需为此设计系统
- 📊 **三层架构**：应用层（提示词、RAG、智能体）、模型开发层（微调）、基础设施层（GPU、向量数据库）；大多数团队应专注于应用层
- 🔧 **适应模型三步骤**：默认顺序为提示工程 → RAG（检索增强生成）→ 微调；不要跳过步骤
- 🤖 **智能体模式**：智能体是循环决策系统，使用 ReAct 模式；默认使用工作流，仅在需要运行时决策时使用智能体
- 📈 **评估优先**：构建评估流水线是首要任务；使用 LLM-as-judge、程序化指标和人工评估的组合；每次模型或提示词变更都必须运行评估
- 🗄️ **RAG 设计模式**：RAG 本质是搜索引擎 + 生成器；混合搜索（BM25+ 语义）加重排序器效果最佳；大多数 RAG 错误是搜索错误
- 💰 **成本管理**：建立成本模型、按功能归因、使用智能路由和缓存；不要对所有请求使用最昂贵的模型
- 🔒 **安全防御**：采用分层防御（输入/输出/智能体级别）；没有纯提示词防御能对抗提示注入
- 🏗️ **生产部署**：从 Agent Engine（托管）开始，需要自定义容器时用 Cloud Run，只有 Cloud Run 不够时才用 GKE
- 📝 **提示词即代码**：将提示词视为 SQL 代码，存放在仓库中，进行版本控制，通过 CI 运行评估；系统提示词应包含稳定指令
