- 🚀 **即时导航核心机制**：通过流式渲染（`<Suspense>`）或缓存（`'use cache'`）将异步操作转为即时可用，实现点击链接后立即显示页面壳层或缓存内容，无需等待网络往返。
- ⚙️ **灵活控制导航行为**：开发者可为特定路由设置 `export const instant = false` 来阻止即时导航（如博客文章），保留服务端驱动的延迟加载特性。
- 🧪 **开发调试工具**：新增“即时洞察”面板和“导航检查器”，可暂停导航至壳层阶段，直观查看预取内容；并提供 Playwright 测试辅助函数 `instant()` 确保即时性不退化。
- 🔄 **预取策略优化**：采用“部分预取”（Partial Prefetching），每个路由仅预取一次可复用壳层（而非每个链接单独请求），减少网络开销，并为离线导航奠定基础。
- 🔧 **配置与兼容性**：需启用 `cacheComponents: true` 和 `partialPrefetching: true` 标志（未来将成为默认行为），支持通过 `<Link prefetch={true}>` 实现更深度的逐链接预取。
- 📈 **实际效果验证**：在 v0 产品中应用后，导航时间显著缩短；提供开源示例“Next Beats”展示即时导航效果。
- ⚠️ **已知问题**：部分阻塞路由可能未被即时洞察报告；Safari 浏览器下工具存在兼容性问题，建议使用 Chrome 或 Firefox 进行开发。

### [使用 – React](https://react.dev/reference/react/use)

**原文标题**: [use – React](https://react.dev/reference/react/use)

React `use` API 是一个用于读取 Promise 或 context 值的 React API，它可以在组件或 Hook 内部调用，并支持在条件语句和循环中使用。

- 📖 **`use(context)` 读取 context 值**：与 `useContext` 类似，但可在 `if` 和 `for` 等条件语句和循环中调用，返回最近的 context provider 提供的值。
- 🌀 **`use(promise)` 读取 Promise 值**：组件会在 Promise 待定时暂停，直到解析后显示数据；若 Promise 被拒绝，则错误会传播到最近的 Error Boundary。
- ⚠️ **使用限制**：`use` 必须在组件或 Hook 内部调用，不能在 `try-catch` 块中使用，且 Promise 必须被缓存以避免重复创建。
- 🗂️ **缓存 Promise**：在客户端组件中，Promise 必须缓存以在重渲染时复用相同实例，避免 Suspense 回退反复显示。
- 🔄 **重新获取数据**：通过 `startTransition` 更新状态中的 Promise 来刷新数据，同时保持现有内容显示。
- 🖱️ **悬停预加载**：在悬停事件中调用缓存函数预加载数据，使 Promise 可能在点击前已解析，从而跳过 Suspense 回退。
- 🌐 **服务端到客户端流式传输**：将 Promise 作为 prop 从服务端组件传递到客户端组件，客户端使用 `use` 读取，支持流式数据加载。
- ❌ **错误处理**：使用 Error Boundary 包裹调用 `use` 的组件，以在 Promise 被拒绝时显示回退 UI，如“重试”按钮。
- 🔧 **常见问题**：避免在 `try-catch` 中使用 `use`（会引发 Suspense 异常），并确保 Promise 被缓存以防止“未缓存 Promise”警告。

### [不同的水化和渲染策略——Nechiu Dan](https://neciudan.dev/hydration-and-rendering-strategies)

**原文标题**: [Different hydration and rendering strategies — Neciu Dan](https://neciudan.dev/hydration-and-rendering-strategies)

本文探讨了不同的水合和渲染策略，旨在缩小从“页面看起来就绪”到“页面真正可交互”之间的差距。

- 📜 **静态生成 (SSG)**：在构建时预渲染 HTML，从 CDN 直接提供，速度极快，但内容可能过时，且大规模构建可能成为瓶颈。
- 🔄 **增量静态再生 (ISR)**：在提供缓存页面的同时，在后台按间隔或按需重新生成页面，兼顾了静态速度与内容的新鲜度。
- 🖥️ **客户端渲染 (CSR)**：服务器发送空壳，浏览器下载并执行所有 JS 来构建 UI。首次加载慢，对 SEO 不友好，但逻辑简单。
- 🖨️ **服务端渲染 (SSR)**：服务器渲染 HTML，浏览器下载 JS 并进行“水合”使其可交互。解决了白屏和 SEO 问题，但存在重复工作和水合不匹配的风险。
- 🌊 **流式 SSR**：使用`Suspense`将 HTML 分块流式传输，用户无需等待最慢的部分即可看到内容，提升了感知加载速度。
- 🏝️ **岛屿架构**：页面大部分为静态 HTML，只对交互部分（岛屿）独立加载和“水合”，大幅减少 JS 体积，适合内容为主的网站。
- 🚀 **React 服务端组件 (RSC)**：组件在服务器运行，只向客户端发送 UI 描述（Flight 流），不发送组件代码，因此无需“水合”，显著减小客户端包体积。
- 🔧 **TanStack Start**：采用客户端优先、按需接入的方式使用 RSC，将 RSC 视为可通过 HTTP 获取的数据流，适合在现有 SPA 中逐步引入。
- ⚡ **细粒度响应式**：SolidJS、Svelte 等框架通过信号（Signal）直接更新特定 DOM 节点，无需虚拟 DOM 和重渲染，使“水合”成本极低。
- 🧘 **可恢复性**：Qwik 框架将服务器执行状态序列化到 HTML 中，客户端无需“水合”，只在用户交互时按需下载和执行相关代码，实现“零水合”。
- 🧩 **Next.js 16.3 即时导航**：通过缓存组件和部分预取，在客户端缓存可复用的页面壳，实现类似 SPA 的即时点击导航体验，同时保留服务端模型优势。
- ✅ **如何选择**：内容网站选静态生成和岛屿架构；应用选 React 服务端组件；特定性能瓶颈再考虑细粒度响应式或可恢复性；登录页可用客户端渲染。

### [追逐幻影跳跃：我们如何在低端安卓设备上让 Pingo 动画流畅运行 — Margelo 博客](https://blog.margelo.com/profiling-skia-reanimated-low-end-android)

**原文标题**: [Chasing a Phantom Jump: How We Made Pingo's Animations Smooth on Low-End Android — Margelo Blog](https://blog.margelo.com/profiling-skia-reanimated-low-end-android)

### 概述总结

本文详细分析了在廉价 120Hz 安卓设备上，一个看似正确的弹簧动画出现视觉卡顿的根本原因。问题并非动画逻辑本身，而是由渲染管线中的多个成本叠加导致帧交付延迟。文章按影响大小依次介绍了三个主要成本：Skia 在安卓上的合成方式（TextureView vs SurfaceView）、一个自循环的重新绘制错误，以及 React 状态管理导致的额外渲染。最终通过将画布从 TextureView 切换到 SurfaceView，并修复其他问题，彻底解决了卡顿。

- 🧩 **核心问题**：动画数学上完美（位置单调），但因帧交付时间不均匀（部分帧耗时超过 8.3ms），导致人眼感知到卡顿。问题在于“帧节奏”，而非“位置曲线”。
- 🖼️ **最大成本：Skia 合成方式**：默认的`TextureView`导致每帧 GPU 需对画布像素进行 2-3 次处理（渲染、上传、合成），在廉价 120Hz 设备上远超 8.3ms 预算。`dumpsys gfxinfo`显示 85% 的帧卡顿，其中 79% 归因于“慢位图上传”。
- 🔬 **诊断方法**：将屏幕强制降至 60Hz 后，卡顿率从 85% 骤降至 15%，证实了每帧 GPU 成本是固定的，且超出了 120Hz 的预算。
- ⚙️ **关键修复**：将`<Canvas>`的`opaque`属性设为`true`（安卓上），使其从`SkiaTextureView`切换为`SkiaSurfaceView`。这为画布创建了独立的`SurfaceFlinger`层，消除了每帧的纹理上传和 GPU 合成，让硬件合成器直接处理，RenderThread CPU 使用率下降 50-65%。
- 🔄 **第二成本：自循环重绘**：`usePathValue`和`usePathInterpolation`钩子会创建一个自循环，每次重绘都重新脏化自身，导致画布在无交互时也持续重绘。通过改用缓冲区变异（`useDerivedValue`返回稳定引用）修复，空闲帧时间从 17-32ms 降至约 9ms。
- ⚡ **第三成本：React 状态搅动**：`useEffect`依赖了每次轮询都创建新对象的 Zustand store 值，导致`React.memo`失效，触发不必要的画布重绘。通过基于内容而非对象引用来记忆化修复。
- 🤖 **自动化二分法**：使用 LLM 代理（Claude Code）并行构建、运行和测量数百个变体，快速定位次要成本。人类负责提出假设和判断结果，机器负责执行重复循环。
- 📊 **最终效果**：在廉价 120Hz 设备上，动画从 85% 卡顿帧变为流畅，用户不再感知到跳跃。所有修复中，`SurfaceView`切换贡献了绝大部分性能提升。
