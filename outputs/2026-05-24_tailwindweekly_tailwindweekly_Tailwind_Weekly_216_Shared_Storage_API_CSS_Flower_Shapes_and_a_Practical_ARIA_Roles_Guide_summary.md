### [ARIA角色详解：Web开发者实用指南](https://www.levelaccess.com/blog/aria-roles-explained-a-practical-guide-for-web-developers/?ref=tailwindweekly.com)

**原文标题**: [ARIA Roles Explained: A Practical Guide for Web Developers](https://www.levelaccess.com/blog/aria-roles-explained-a-practical-guide-for-web-developers/?ref=tailwindweekly.com)

### 概述总结  
ARIA 角色为网页元素添加语义信息，帮助辅助技术（如屏幕阅读器）理解页面结构和交互。正确使用时，ARIA 角色能提升无障碍性；但误用可能导致障碍。开发者应优先使用原生HTML元素，仅在必要时应用ARIA角色，并确保测试和验证。

- 📚 **ARIA角色定义**：为`div`或`span`等非语义元素提供语义含义，使辅助技术能正确解释自定义组件和复杂布局。
- 🌳 **无障碍树机制**：浏览器通过无障碍树将ARIA角色传递给辅助技术，而非直接读取DOM。
- 🚫 **优先使用原生HTML**：当存在语义HTML元素（如`<button>`）时，应避免使用ARIA角色，以减少不一致性和额外属性需求。
- ⚠️ **误用风险**：错误使用ARIA角色（如重复角色、无效值）可能导致屏幕阅读器忽略元素或提供误导信息。
- 🧩 **角色、状态与属性**：ARIA角色定义元素类型（如按钮），状态（如`aria-expanded`）描述当前条件，属性（如`aria-labelledby`）提供额外上下文。
- 🏷️ **六大角色类别**：包括地标角色（如导航）、文档结构角色、小部件角色、复合小部件、活动区域角色和窗口角色，每类满足不同无障碍需求。
- ✅ **无障碍标准**：ARIA不是WCAG合规的强制要求，但SC 4.1.2（名称、角色、值）要求元素向辅助技术暴露正确信息，ARIA可在此场景下使用。
- ❌ **常见错误**：避免在已有语义的元素上添加ARIA角色、使用无效角色值、过度使用地标角色，或使用抽象角色（如`role=”widget”`）。
- 🛠️ **测试方法**：使用NVDA（Windows）或VoiceOver（macOS）等屏幕阅读器，验证地标导航、小部件键盘交互和活动区域更新。
- 📖 **FAQ要点**：语义HTML通常无需ARIA；ARIA角色非法律强制，但WCAG 4.1.2可能间接要求；抽象角色不可用于HTML。
