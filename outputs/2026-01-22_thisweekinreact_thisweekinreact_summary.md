### [使用视图过渡类型 - Web API | MDN](https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API/Using_types)

**原文标题**: [Using view transition types - Web APIs | MDN](https://developer.mozilla.org/en-US/docs/Web/API/View_Transition_API/Using_types)

视图过渡类型允许为不同的过渡场景指定类型，并通过 CSS 为 DOM 元素内容更新应用不同的动画效果，例如在图片库中根据前进、后退、添加或删除图片等操作应用不同的动画。

- 🎨 视图过渡类型通过`Document.startViewTransition()`的`types`参数或`@view-transition`规则设置，CSS 中可用`:active-view-transition-type()`为不同类型应用定制动画
- 🔄 单页面应用（SPA）中，通过`startViewTransition()`的`types`数组指定类型（如"forwards"、"backwards"），并在 CSS 中为不同类型配置`::view-transition-old/new()`动画
- 🌐 多页面应用（MPA）中，通过`@view-transition`规则的`types`描述符设置类型，或利用`pageswap`/`pagereveal`事件动态修改`ViewTransition.types`属性
- ⚙️ 可通过`ViewTransitionTypeSet`的方法（如`add()`、`delete()`）动态调整过渡类型，实现基于导航方向（如前/后章节）的自适应动画
- 📁 示例包括 SPA 图片库（前进/后退/点击缩略图不同动画）和 MPA 故事应用（章节间导航的定向滑动动画），完整代码可在 MDN 示例仓库查看

- 📜 **根本弱点**：依赖“栈空间耗尽后可恢复”这一未规范定义的行为（CWE-758）来保证服务可用性存在风险。即使修复后，应用程序和框架仍应避免依赖此行为进行安全防御。
- 🚨 **行动建议**：用户应立即升级到已修复版本；库和框架应实施更健壮的防御（如限制递归深度），而非依赖运行时抛出可捕获错误。

**原文标题**: [useRender · Base UI](https://base-ui.com/react/utils/use-render#migrating-from-radix-ui)

- 🛠️ **核心功能**：`useRender` 允许构建支持 `render` 属性的自定义组件，用于替换默认的 HTML 元素或组件。
- 📝 **基本用法**：通过 `defaultTagName` 设置默认标签，`render` 属性可接受 React 元素或回调函数来定制渲染。
- 🔄 **回调函数**：`render` 的回调版本可接收组件内部状态（`state`），实现更动态的渲染逻辑。
- 🧩 **属性合并**：使用 `mergeProps` 函数安全合并事件处理器、`className` 和 `style` 等属性。

- 📘 **TypeScript 支持**：提供 `useRender.ComponentProps` 和 `useRender.ElementProps` 接口来类型化组件属性。

- 📚 **API 参考**：`useRender` 接受 `defaultTagName`、`render`、`props`、`ref`、`state` 等参数，返回渲染的 React 元素。

### [你能用 React 服务器操作获取数据吗？](https://www.developerway.com/posts/server-actions-for-data-fetching)

**原文标题**: [Can You Fetch Data with React Server Actions?](https://www.developerway.com/posts/server-actions-for-data-fetching)

本文探讨了是否可以使用 React Server Actions（现称 Server Functions）进行客户端数据获取。虽然技术上可行，但存在性能问题，特别是并行请求会变为串行，导致数据加载时间显著增加。文章建议在客户端数据获取时仍使用传统的 REST 请求配合数据获取库（如 TanStack Query），而非 Server Actions。

- 🧐 **技术可行性**：Server Actions 本质上是 POST 请求的封装，可用于数据获取，提供类型安全和 IDE 自动补全。
- 🚫 **性能缺陷**：使用 Server Actions 获取数据时，并行请求会变为串行，显著延长加载时间（例如从 1.7 秒增至 8 秒）。
- 📜 **文档说明**：React 文档明确指出，Server Actions 通常一次处理一个操作，不适合并行数据获取。
- 🔍 **调试困难**：Network 面板中所有请求名称相同，响应格式不易阅读，增加调试复杂度。
- 🛡️ **缓存限制**：POST 请求无法通过 HTTP 头部进行浏览器或 CDN 缓存，但客户端缓存仍可工作。
- 🔄 **适用场景**：Server Actions 适合数据变更操作（如提交表单），而非数据查询。
- ⚖️ **架构建议**：客户端数据获取应使用 REST 请求配合数据获取库，Server Components 更适合服务端渲染场景。
- 📊 **性能对比**：文章通过仪表板应用对比了不同数据获取方式的性能，显示 Server Actions 在并行请求时表现最差。

- 🌐 **请求中间件应用**：适用于所有服务器请求（包括路由和 SSR），可通过 `.server()` 方法定义服务器端逻辑，并灵活配置到特定或全部路由方法。

- 📡 **请求与响应定制**：可读取或修改服务器响应头、状态码，并通过客户端中间件设置自定义请求头，支持多中间件头合并与覆盖。
- 🔧 **自定义 Fetch 实现**：允许通过中间件或调用点提供自定义 `fetch` 实现，用于添加拦截器、重试逻辑或监控，优先级为调用点最高。
- 🌳 **环境与性能优化**：中间件代码根据环境进行 Tree Shaking，服务器端保留全部代码，客户端移除服务器专用部分，提升性能。
