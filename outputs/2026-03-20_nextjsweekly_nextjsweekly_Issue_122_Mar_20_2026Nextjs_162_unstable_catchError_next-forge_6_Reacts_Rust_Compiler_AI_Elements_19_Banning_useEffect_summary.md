**原文标题**: [Functions: unstable_catchError | Next.js](https://nextjs.org/docs/app/api-reference/functions/catchError)

Next.js 的 `unstable_catchError` 函数是一个用于在组件树中创建错误边界以捕获和处理错误的程序化 API。

- 🛡️ **创建错误边界**：该函数通过包装子组件来创建一个错误边界，为 `error.js` 文件约定提供了组件级的替代方案。
- ⚙️ **参数与用法**：它接受一个回退函数作为参数，该函数在捕获错误时负责渲染错误界面，并接收相关错误信息和恢复函数。
- 🔄 **错误恢复机制**：提供了 `retry()` 函数来尝试重新获取和渲染内容以从错误中恢复，通常比仅重置错误状态的 `reset()` 函数更有效。
- 🧩 **灵活性与范围**：与路由段范围的 `error.js` 不同，它可以包装组件树的任何部分，实现细粒度的错误处理，并且支持传递服务器渲染的内容作为回退 UI。
- 📝 **注意事项**：回退函数必须是客户端组件；某些使用模式可能导致回退内容在每次渲染时都被预先渲染。

### [开发者不喜却无法回避的两个 React 设计选择 - DEV 社区](https://dev.to/playfulprogramming/two-react-design-choices-developers-dont-like-but-cant-avoid-d6g)

**原文标题**: [Two React Design Choices Developers Don’t Like—But Can’t Avoid - DEV Community](https://dev.to/playfulprogramming/two-react-design-choices-developers-dont-like-but-cant-avoid-d6g)

本文探讨了 React 中两个备受争议的设计选择——延迟状态提交和 Effect 依赖数组，并指出这些并非 React 的任意决策，而是异步 UI 系统中不可避免的约束。作者通过开发 Solid 2.0 的经验，认识到异步操作迫使框架必须隔离提交、提前知晓 Effect 依赖，以保持 UI 的一致性。这些是任何处理异步的框架都必须面对的基本规则，而非 React 独有的问题。

- 🧠 **异步操作迫使状态提交必须隔离**：在异步场景下，若衍生值未就绪，状态更新必须延迟提交，否则会导致 UI 与数据不一致，破坏确定性。
- 📋 **Effect 的依赖必须在计算时已知**：为了在异步环境中保证副作用执行的确定性，必须在执行前收集所有依赖，避免因异步解析时间不同而引发不可预测的行为。
- ⚙️ **信号（Signals）无法完全解决异步问题**：信号虽能同步更新状态与衍生值，但一旦衍生值变为异步，其同步保证即失效，仍需面对延迟提交的需求。
- 🔧 **编译器无法根本解决异步语义问题**：静态分析无法动态判断值的真实性与推测性，异步是运行时现象，必须在运行时层面保证一致性。
- 🚀 **接纳约束是框架成熟的标志**：Solid 2.0 选择内化这些约束，并非模仿 React，而是承认异步是 UI 架构中的一等公民，从而构建出既高效又正确的系统。
