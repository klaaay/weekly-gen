### [红鲱鱼与 AI 垃圾：揭穿 React2Shell 的误导信息](https://miggo.io/post/red-herrings-and-ai-slop-debunking-react2shell-misinformation)

**原文标题**: [Red Herrings and AI Slop: Debunking React2Shell Misinformation](https://miggo.io/post/red-herrings-and-ai-slop-debunking-react2shell-misinformation)

本文分析了 React2Shell 漏洞（CVE-2025-55182）的真相与误解，揭示了社区和 AI 在漏洞分析中的误导，并明确了实际的攻击路径和影响范围。

- 🚨 **漏洞本质**：React2Shell 是一个影响 React Server Components（RSC）的远程代码执行漏洞，攻击通过 multipart/form-data 载荷和特定 Flight 协议操作符（$@、$B、$n）触发。
- 🎯 **攻击条件**：任何处理 RSC 请求的服务器均受影响，即使未定义 Server Functions；传统 SSR 设置不受影响。
- ❌ **常见误解**：社区错误地认为$F原语和 loadServerReference 是攻击路径，但实际这些是干扰项；许多 PoC 因模拟内部服务器行为而无效。
- 🤖 **AI 误导**：AI 加速了错误分析，生成不切实际的漏洞利用代码，导致研究资源浪费，突显了验证 AI 结论的重要性。
- 🔍 **技术细节**：漏洞在 RSC 请求处理的早期阶段触发，早于服务器函数验证；React 通过约 700 行的混淆补丁延缓了漏洞分析。
- 📊 **影响范围**：仅影响处理 RSC 的服务器，标准 SSR 应用安全；攻击无需已知服务器函数 ID。
- ⚠️ **安全启示**：强调在漏洞响应中需区分 AI 推测与已验证行为，避免依赖未经验证的社区分析。

**原文标题**: [Do's and Don'ts of useEffectEvent in React](https://slicker.me/react/useEffectEvent.html)

useEffectEvent 是一个实验性的 React Hook，用于从 Effect 中提取非响应式逻辑，允许在 Effect 内部读取最新的 props 或 state 值，而不会因这些值的变化触发 Effect 重新执行。

- 🧩 **解决核心问题**：传统 Effect 中，若需读取最新值（如主题），必须将其加入依赖数组，导致不必要的重新执行；useEffectEvent 创建稳定的函数引用，始终读取最新值但避免 Effect 重复运行。
- ⚙️ **正确用法**：在 Effect 内部直接调用、用于读取最新 props/state 而无需添加依赖、结合清理逻辑使用，或处理需访问 Effect 上下文的回调。
- 🚫 **错误用法**：避免在常规事件处理器、渲染过程中、异步调用或延迟后使用，也不应将其作为 prop 传递给组件或替代正确的记忆化。
- 📊 **常见用例**：适用于日志记录与分析（如跟踪页面访问）、需最新状态的回调（如第三方库集成），以及基于最新值的防抖操作。
- 🛠️ **最佳实践**：仅在确实需要读取非响应式值时使用，保持函数目的单一，并等待稳定版发布后再用于生产环境。

- 🎯 **核心概念**：受控与不受控组件的本质区别在于状态所有权——受控由父组件通过 props 控制，不受控由组件内部管理。
- 📝 **表单输入示例**：受控输入使用`value`和`onChange`由 React 状态管理；不受控输入使用`defaultValue`由 DOM 管理初始值。
- ⚙️ **组件设计模式**：自定义组件中，受控模式通过 props（如`isOn`和`onToggle`）让父组件控制状态；不受控模式使用内部状态（如`useState`）。
- 🔄 **双模式支持**：组件可同时支持受控和不受控用法，通过检查 props（如`isOn !== undefined`）决定模式，但应避免运行时切换。
- 🌍 **实际应用**：常见 UI 组件（如标签页、模态框、手风琴、日期选择器）都遵循此模式，受控组件便于多组件协调与复杂行为实现。
- ⚖️ **权衡考虑**：选择取决于父组件是否需要知晓或影响状态；不受控通常更简单，受控则支持实时验证和协调。
- 📚 **关键总结**：使用`useState`不意味组件受控；从父组件视角，若无法通过 props 影响状态，组件即为不受控。

### [缓解 React 服务器组件中的关键安全漏洞 - Expo 更新日志](https://expo.dev/changelog/mitigating-critical-security-vulnerability-in-react-server-components)

**原文标题**: [Mitigating the Critical Security Vulnerability in React Server Components - Expo Changelog](https://expo.dev/changelog/mitigating-critical-security-vulnerability-in-react-server-components)

React Server Components 存在一个严重的远程代码执行安全漏洞（CVE-2025-55182），仅影响使用实验性 RSC 或 Server Functions 的 Expo 应用。用户需根据其 Expo SDK 版本，将 `react-server-dom-webpack` 依赖升级到特定安全版本以进行修复，或安装相应的 Expo 补丁包。升级后应验证依赖版本，并注意 React 与 React Native 的版本兼容性。

- 🚨 **漏洞概述**：React Server Components 中存在一个未经身份验证的远程代码执行漏洞（CVE-2025-55182），仅影响使用实验性 RSC 或 Server Functions 的 Expo 应用。
- 📦 **影响范围**：仅当项目依赖 `react-server-dom-webpack` 的 19.0、19.1.0、19.1.1 或 19.2.0 版本时可能受影响；API 路由不受影响。
- 🔧 **修复措施**：根据 Expo SDK 版本升级 `react-server-dom-webpack` 至指定安全版本（如 SDK 54 对应 19.1.2，SDK 53 对应 19.0.1）。
- 🛡️ **官方补丁**：Expo 已为 SDK 53、54 和 canary 发布补丁，通过更新 `jest-expo` 和 `expo-router` 来限制使用安全版本。
- 📋 **升级验证**：升级后使用 `npm explain react-server-dom-webpack` 等命令验证依赖版本是否正确。
- ⚠️ **版本兼容**：注意 `react-native` 需要特定版本的 `react`，避免运行时版本不匹配；可使用 `expo install --check` 检查推荐版本。
- 🏗️ **Monorepo 支持**：在 monorepo 中，可通过配置 `app.json` 使 Expo 应用与其他项目使用不同的 React 版本。

Chrome 144 Beta 版本引入了多项 CSS、Web API 及功能更新，同时宣布了多项 API 的弃用计划。

- 🎨 **CSS 锚点定位支持变换**：当锚点元素应用了变换时，`anchor()` 和 `anchor-size()` 函数将基于变换后的边界框进行解析。
- 🔍 **CSS 查找高亮伪元素**：新增 `::search-text` 伪元素，允许自定义查找结果的样式，提升对比度和视觉体验。
- 🚫 **弃用 aria-details 用于锚点定位**：Chrome 将不再为非弹出式锚点定位场景创建 `aria-details` 关系。
- 📜 **@scroll-state 滚动方向支持**：新增 `scrolled` 状态，允许根据滚动方向对容器后代进行样式控制。
- ⌨️ **键盘滚动遵守 overscroll-behavior**：确保键盘滚动与鼠标/触摸滚动一致，遵守 `overscroll-behavior` 设置以防止滚动链。
- 🌳 **非树作用域的 container-name 匹配**：`@container` 查询中的 `container-name` 匹配不再受树作用域限制，提升了跨树（如影子 DOM）的样式灵活性。
- 🖼️ **SVG2 CSS 级联更新**：CSS 规则现在匹配 `<use>` 元素的实例化元素，而非原始子树，使状态选择器（如 `:hover`）能在实例中生效。
- 🕶️ **新增 XRVisibilityMaskChange 事件**：提供视口可见区域的网格数据，用于优化 WebXR 应用的绘制性能。
- ⚡ **View Transitions 新增 waitUntil() 方法**：允许开发者通过 Promise 延迟过渡伪元素树的销毁，支持与滚动驱动动画等高级用例结合。
- 📍 **引入 <geolocation> 元素**：提供声明式、用户激活的位置访问控件，简化权限流程并改善用户体验。
- 📋 **新增 clipboardchange 事件**：当系统剪贴板内容变化时触发，便于 Web 应用（如远程桌面）同步剪贴板状态。
- 🚨 **弃用多项隐私沙盒 API**：包括 Topics API、Protected Audience API、Shared Storage API、Attribution Reporting API、Private Aggregation API、Related Website Sets 及 `requestStorageAccessFor`，因 Chrome 决定维持现有第三方 Cookie 策略。
- ⚙️ **同步标签组策略调整**：桌面端 `savedTabGroups` 作为独立同步数据类型被弃用，其功能已整合至 `tabs` 数据类型中。
- ⚠️ **停止加载外部 XML 实体**：Chrome 将逐步停止在非 XSLT 的 XML 解析中同步加载外部实体或 DTD，以提升安全性与一致性。

### [Bun v1.3.4 | Bun 博客](https://bun.com/blog/bun-v1.3.4)

**原文标题**: [Bun v1.3.4 | Bun Blog](https://bun.com/blog/bun-v1.3.4)

本文介绍了 Bun 的最新更新，包括安装与升级方法、新增功能如 URLPattern API 和测试假定时器、fetch() 代理头自定义，以及多项错误修复与性能改进。

- 🚀 **安装与升级**：支持多种安装方式（curl、npm、PowerShell 等），升级命令为 `bun upgrade`
- 🔗 **URLPattern API**：新增 URL 模式匹配 Web API，支持路由匹配和参数提取，适用于 Web 服务器
- ⏱️ **测试假定时器**：`bun:test` 支持假定时器，可控制时间测试定时相关代码，无需真实等待
- 🌐 **自定义代理头**：`fetch()` 代理选项现支持对象格式，可添加自定义头部如代理认证信息
- 🔧 **连接池修复**：修复 `http.Agent` 在 `keepAlive: true` 时未重用连接的多个错误
- ⚡ **独立执行文件优化**：编译后的可执行文件默认不加载运行时配置文件，提升启动性能

- 🗃️ **SQLite 更新**：`bun:sqlite` 升级至 v3.51.1，修复查询优化问题

### [类型感知 Linting Alpha | JavaScript 氧化编译器](https://oxc.rs/blog/2025-12-08-type-aware-alpha.html)

**原文标题**: [Type-Aware Linting Alpha | The JavaScript Oxidation Compiler](https://oxc.rs/blog/2025-12-08-type-aware-alpha.html)

Oxlint 宣布其类型感知 linting 功能进入 Alpha 阶段，该功能利用 TypeScript 类型系统实现强大的代码检查规则，现已包含 43 条规则，能显著提升代码质量并捕获潜在运行时错误。相较于 ESLint，其性能提升约 10 倍，并新增了类型检查、规则配置、内联禁用注释支持以及自动修复等特性。

- 🚀 **快速开始**：通过安装 `oxlint` 和 `oxlint-tsgolint` 并使用 `--type-aware` 标志，即可在几分钟内启用类型感知 linting。
- ⚡ **卓越性能**：基准测试显示，Oxlint 的类型感知 linting 比 ESLint 配合 typescript-eslint 快约 10 倍，并能复用类型信息进行类型检查，避免重复工作。
- 🆕 **新增功能**：自技术预览版以来，增加了在 linting 同时进行类型检查、完整的规则配置支持、内联禁用注释以及自动修复等功能。
- 🏗️ **技术架构**：采用独特的双二进制架构（Rust 编写的 `oxlint` CLI 和 Go 编写的 `tsgolint`），在保持核心速度的同时利用 TypeScript 类型系统。
- ⚠️ **注意事项**：基于 TypeScript 7.0+，不支持旧版语法和已弃用的 `tsconfig.json` 选项（如 `baseUrl`），在大型代码库中可能遇到内存问题。
- 🔮 **未来计划**：Beta 版本将致力于支持更多规则（目标覆盖 typescript-eslint 的 59 条规则）、优化性能与内存使用，并解决已知问题。

### [Firefox 146 开发者版本发布说明（稳定版）- Mozilla | MDN](https://developer.mozilla.org/en-US/docs/Mozilla/Firefox/Releases/146)

**原文标题**: [Firefox 146 release notes for developers (Stable) - Mozilla | MDN](https://developer.mozilla.org/en-US/docs/Mozilla/Firefox/Releases/146)

Firefox 146 版本于 2025 年 12 月 9 日发布，为开发者带来了多项更新，包括开发者工具、CSS、JavaScript、API 及 WebDriver 的改进，同时引入了实验性功能如导航 API 和自定义媒体查询。

- 🔧 **开发者工具优化**：检查器的规则视图中，若 CSS 自定义属性未使用且数量超过 10 个，默认隐藏以减少干扰，可通过“显示...”按钮查看。
- 📐 **MathML 增强**：修复了 RTL 模式下运算符镜像与拉伸的组合问题，并新增支持`math-shift`属性以控制上标渲染的紧凑度。
- 🎨 **CSS 功能扩展**：新增`contrast-color()`函数确保颜色对比度达标，支持`display-p3-linear`色彩空间提升精度，引入`text-decoration-inset`调整文本装饰位置，默认支持`@scope`规则精准选择 DOM 子树，并兼容`-webkit-fill-available`关键字改进网页兼容性。
- ⚙️ **JavaScript 更新**：`WeakMap`和`WeakSet`现在接受除已注册外的`Symbol`对象作为键。
- 🔑 **API 改进**：`SubtleCrypto.importKey()`支持导入压缩椭圆曲线点定义的密钥，用于 ECDSA 和 ECDH 算法。
- 🌐 **WebDriver 升级**：更新了区域设置和时区覆盖命令，新增上下文定位器以检索非顶级浏览上下文容器，实现`network.setExtraHeaders`命令添加请求头，并优化网络数据收集功能。
- 🛠️ **附加组件开发调整**：`browsingData.removeLocalStorage`及相关方法现在同时删除`sessionStorage`中的对象。
- 🧪 **实验性功能引入**：包括`<meta name="rating">`元素限制成人内容、导航 API 支持单页应用管理、CSS `shape()`曲线命令中的相对控制点，以及`@custom-media`规则简化媒体查询。
