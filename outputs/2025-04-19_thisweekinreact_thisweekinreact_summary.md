**原文标题**: [Avoid the State Synchronization Trap | Ondrej Velisek](https://ondrejvelisek.github.io/avoid-state-synchronization-trap/)

文章讨论了状态管理中常见的状态同步陷阱，通过一个订单表格的示例展示了问题及其解决方案。

- 🚨 **问题概述**：状态同步（如列与筛选器）常导致错误、卡顿和性能问题，通常源于状态形状设计不当。
- 🔄 **解决方案**：通过拆分状态并在选择器/渲染函数中合并，避免直接同步。例如，将可见属性提取为共享状态，派生列和筛选器。
- ⏳ **时间线示例**：
  - 2025 年：简单订单列表，使用`useOrders`获取数据。
  - 2026 年：支持自定义列，通过上下文管理列状态。
  - 2027 年：添加筛选功能，需同步列与筛选器状态，引发初始化问题。
- ⚠️ **同步陷阱**：
  - **直接更新**：需手动同步多处逻辑，易遗漏（如重置列时忘记重置筛选器）。
  - **包装函数**：集中更新逻辑，但可能遗漏合并或引发循环依赖。
  - **副作用（useEffect）**：异步更新导致 UI 闪烁和重复请求。
- ✅ **推荐方案**：**选择器模式**。通过派生状态（如`useFilters`钩子）同步数据，避免重复状态，提升可维护性。
- 📌 **结论**：状态同步是代码异味，优先通过选择器解决。若需保留，用包装函数 + 校验逻辑，避免`useEffect`。

### ["你可能不需要 react-native-svg | Software Mansion"](https://blog.swmansion.com/you-might-not-need-react-native-svg-b5c65646d01f?gi=a0accc08e20d)

**原文标题**: [You Might Not Need react-native-svg  | Software Mansion](https://blog.swmansion.com/you-might-not-need-react-native-svg-b5c65646d01f?gi=a0accc08e20d)

react-native-svg 是一个历史悠久的库，用于在 React Native 中渲染 SVG 图像。文章探讨了其优缺点、替代方案以及适用场景。

- 🏗️ **react-native-svg 简介**：该库提供两种渲染 SVG 的方式，一种是通过 React 组件直接控制 SVG 元素，另一种是通过 `SvgXml` 或 `LocalSvg` 直接解析 SVG 字符串。
- ⚙️ **原生实现**：iOS 使用 `CGContext`，Android 使用 Canvas 和 Bitmap，但需要手动实现 SVG 标准的所有元素和行为。
- 🚫 **缺点与限制**：包括不必要的 React 组件树、无默认记忆化、内存泄漏问题（尤其在 iOS 上），以及每次操作都会重新绘制。
- 🔄 **替代方案**：
  - 🖼️ **Expo Image**：使用 Glide（Android）和 SDWebImage（iOS）渲染 SVG，但异步加载可能导致闪烁，且不支持动画。
  - 🎬 **Lottie 和 Rive**：适用于动画需求，性能优于 SVG。
  - 🎨 **react-native-skia**：支持大部分 SVG 标准，但内存开销较大。
  - 🌐 **WebView**：浏览器渲染 SVG 效果最佳，适合特定场景。
  - ✒️ **字体图标**：如 Font Awesome，性能优异且易于使用。
  - 📦 **react-native-vector-image**：将 SVG 转换为原生资源（Android 的 Vector Drawables 和 iOS 的 PDF）。
- ✅ **何时使用 react-native-svg**：当需要对 SVG 各部分进行动态控制或动画处理时，它是理想选择。
- 🙏 **致谢**：感谢社区和贡献者的支持，特别是 Horcrux、Mikael Sand 和 Jakub Grzywacz 等人的工作。

### ["RollDown 工作原理：模块加载、依赖图与优化详解 | Atriiy"](https://www.atriiy.dev/blog/rolldown-module-loader-and-dependency-graph)

**原文标题**: [How Rolldown Works: Module Loading, Dependency Graphs, and Optimization Explained | Atriiy](https://www.atriiy.dev/blog/rolldown-module-loader-and-dependency-graph)

Rolldown 是一个基于 Rust 开发的超快 JavaScript 打包工具，旨在与 Rollup API 无缝协作，并计划成为 Vite 的统一打包工具。其核心优势在于性能（比 Rollup 快 10~30 倍）和开发/生产环境的一致性。以下是关键要点：

- 🚀 **核心目标**：替代 Vite 现有的 esbuild（开发） + Rollup（生产）双打包方案，提供单一工具链和更一致的构建结果。
- 🔍 **工作流程**：分为依赖图构建、优化、代码生成/打包、输出四个阶段，模块加载器（ModuleLoader）是扫描阶段的核心组件。
- 📦 **模块加载**：负责定位、获取和解析模块（JS/CSS/资源等），转换为内部数据结构，支持异步并发处理和缓存优化。
- 🧩 **数据结构**：
  - 使用 `ModuleIdx` 轻量索引替代模块引用，提升性能。
  - `HybridIndexVec` 根据场景选择紧凑向量或稀疏哈希存储。
  - `ScanStageCache` 支持增量构建，通过反向依赖表快速定位变更影响范围。
- 🌐 **依赖图**：
  - 采用邻接表表示模块关系，高效处理稀疏连接。
  - 同时构建正向（当前模块的依赖）和反向（依赖当前模块的模块）图，优化树摇和增量构建。
- ⚡ **性能优化**：
  - 基于 Tokio 的异步并发任务处理模块。
  - 多级缓存（如模块 ID 映射、快照复用）减少重复计算。
- 🔮 **未来计划**：持续开发以完全适配 Vite，详见[官方路线图](https://rolldown.rs/roadmap)。
