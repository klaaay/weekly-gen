### [谁拥有这棵树？RSC作为协议而非架构 | TanStack博客](https://tanstack.com/blog/who-owns-the-tree)

**原文标题**: [Who Owns the Tree? RSC as a Protocol, Not an Architecture | TanStack Blog](https://tanstack.com/blog/who-owns-the-tree)

TanStack Start 支持两种 RSC 组合模型：服务端拥有树和客户端拥有树，并引入 Composite Components 来填补客户端树中嵌入服务端渲染内容的空白。

- 🧩 RSC 本质是协议而非架构：它定义了序列化 React 输出、客户端引用和非 JSON 数据的流式传输方式，而“服务端拥有树”只是其中一种使用方式。
- 🔄 两种模型可相互转化：服务端模型通过 `'use client'` 可变为 SPA；客户端模型通过 Composite Components 可嵌入服务端渲染内容，两者都基于同一协议。
- 📊 实际案例：客户端主导的仪表盘只需通过 `createFromReadableStream` 获取服务端渲染的图表，无需将整个路由转为服务端架构。
- ⚖️ 框架倾向性：服务端模型自然拉向服务端，客户端模型更自然拉向客户端，但两者都能达到对方极端，只是阻力不同。
- 🧪 Composite Components 实现控制反转：服务端函数返回 Flight 数据，客户端通过 `<CompositeComponent>` 接收并嵌入子节点，实现客户端树中的服务端渲染片段。
- 🚫 不提供缓存指令：因 TanStack Start 支持多种部署平台（Cloudflare、Netlify、Node 等），无法假设统一持久化层，改用透明字节流在各层（渲染、服务端、网络、客户端）自行缓存。
- 🎯 核心哲学：框架应提供灵活的工具组合，而非强制 RSC 作为银弹。开发者最了解应用需求，框架应让路。

- 📊 **大規模靜態生成的真實基準**：從 1,000 頁到 500,000 頁，測試了 Next.js、TanStack Start、Gatsby、Astro 和 @lazarv/react-server，結果顯示大多數框架在 1,000 頁時表現相近，但超過 100,000 頁後差異巨大。
- ⏱️ **TTFP 曲線揭示關鍵設計差異**：@lazarv/react-server 的「首頁生成時間」幾乎是平穩的直線（1.4–3.2 秒），而其他框架則急劇上升，Next.js 在 100,000 頁時高達 124 秒。

- 📦 **輸出大小差異懸殊**：100,000 頁時，Astro 僅 47 MiB，@lazarv/react-server 83 MiB，而 Next.js 高達 1.84 GiB（因包含 RSC 有效載荷和運行時捆綁）。
- 🧠 **記憶體使用對比**：Gatsby 在 500,000 頁時峰值 RSS 達 9.55 GiB，而 @lazarv/react-server 僅 2.6 GiB，且曲線平穩。
- 🚀 **Astro 最快但非 React 運行時**：Astro 使用自有模板語言，@lazarv/react-server 則在運行 React Server Components 的情況下，效能與 Astro 接近（100,000 頁：26 秒 vs 22.6 秒）。
- ⚖️ **ISR 並非靜態部署的替代方案**：ISR 需要運行時，不適用於純靜態 CDN（如 GitHub Pages、S3），且會將構建成本轉嫁為每次請求的計算成本。
- 🛠️ **選擇 SSG 的關鍵建議**：若路線圖超過 10,000 頁，應優先檢查路徑列表 API 是「返回陣列」還是「流式生成」——這決定了能否大規模擴展。
- 📂 **基準測試開源可複現**：完整程式碼在 GitHub，包含 `apps/`、`bench/` 和 `bench/REPORT.md`，歡迎提交 PR 改進。

- 🧩 **问题背景**：直接使用Ant Design的Button组件传递`href`属性会导致全页浏览器导航而非Next.js路由导航，影响用户体验。
- ⚠️ **方法一：命令式路由**：通过`useRouter().push()`实现路由跳转，但会渲染`<button>`元素而非语义化的`<a>`链接，存在可访问性问题。

- 🔮 **未来风险**：`legacyBehavior`属性可能在未来Next.js版本中被移除，需关注官方更新。
- 🛠️ **可复用组件**：创建`RouterButton`组件封装上述逻辑，支持传递`href`、`linkProps`（Next链接属性）和Ant Button的其他属性，简化使用。

### [第二部分：通过更高效的数据结构加速JSI](https://blog.margelo.com/make-jsi-run-faster-2)

**原文标题**: [Part 2: Making JSI Faster with more Efficient Data Structures](https://blog.margelo.com/make-jsi-run-faster-2)

### 概述总结
本文深入探讨了在JSI模块中，数据表示方式对性能的巨大影响。通过多个基准测试，展示了从C++向JavaScript传递数据时，选择不同的数据结构（对象数组、扁平数组、ArrayBuffer）会导致高达30倍的性能差异。同时，文章还分析了API形状（字符串vs数字）、字符串构建（std::format、字符串拼接、char缓冲区）以及数字转字符串（std::to_string vs std::to_chars）等场景下的性能优化策略。核心结论是：性能瓶颈往往不在于算法，而在于数据表示和跨层边界调用的次数。

- 📊 **数据形状是关键**：从C++返回点坐标列表时，使用`Array<Object>`（1036.50ms）比使用`ArrayBuffer`（34.81ms）慢约30倍。扁平数组（311.78ms）作为中间方案，性能提升3.3倍。
- 🧠 **减少JSI边界调用**：`Array<Object>`的性能瓶颈在于每个`jsi::Object`的独立分配和多次`setProperty`调用，这些都会增加GC压力和跨层开销。
- ⚡ **ArrayBuffer实现零拷贝**：通过`jsi::MutableBuffer`将C++的`std::vector`内存直接暴露给JavaScript的`ArrayBuffer`，避免了元素级复制和JS对象分配，是处理大量数值型数据的最优方案。
- 🔢 **API形状：用数字代替字符串**：在需要根据类型参数分支时，使用数字索引（`asNumber()`）和查找表（8.67ms）比使用字符串比较（12.88ms）快1.5倍，且代码更简洁。
- 🔧 **字符串构建：char缓冲区最快**：在热路径上构建URL时，使用栈上`char`缓冲区（7.78ms）比`std::format`（21.35ms）快2.7倍，比`std::string`拼接（12.38ms）快1.6倍，但需注意安全性。
- 🔢 **数字转字符串：std::to_chars更高效**：在频繁的数字转字符串场景中，`std::to_chars`（9.75ms）比`std::to_string`（16.27ms）快1.7倍，因为它避免了临时字符串的分配和复制。
- 🚀 **性能提升是数量级**：通过选择合适的数据表示，性能提升不是10-20%，而是数量级（如30倍），这在摄像头处理等实时场景中至关重要。
- 💡 **优化原则**：最大的性能收益往往来自减少边界调用次数和选择正确的数据形状，而非复杂的算法。

**原文标题**: [
  The end of responsive images - Piccalilli
](https://piccalil.li/blog/the-end-of-responsive-images/)

### 概述总结
- 🖼️ 响应式图片的 `sizes` 属性曾是开发者噩梦，但新引入的 `sizes="auto"` 值配合 `loading="lazy"` 将彻底终结这一痛苦。
- 👑 作者马特·马奎斯作为 RICG 主席，承认自己当年主导了复杂且令人憎恨的 `sizes` 语法标准化，但强调这是当时唯一可行的方案。
- ⚖️ 描述性语法（`srcset`/`sizes`）让浏览器根据上下文自主选择最佳图片源，避免了开发者因过度控制而导致的性能与体验问题。
- 🚫 `sizes` 属性因依赖布局信息而难以自动化，开发者被迫手动编写复杂条件表达式（如 `(min-width: 1340px) 257px`），成为开发痛点。
- 💡 新方案 `sizes="auto"` 仅对懒加载图片生效，浏览器可在渲染后自动计算图片实际尺寸，无需手动指定复杂规则。

- 🎯 仅首屏关键图片（如英雄图）仍需手动设置 `sizes`，其余图片均可通过 `loading="lazy" sizes="auto"` 实现全自动优化。
- 📉 图片占网站传输量绝大部分，响应式图片机制通过让浏览器自主决策，为全球用户节省了难以估量的带宽。

### [看似整洁代码的O(n²)性能缺陷 - DEV社区](https://dev.to/aralroca/the-on2-bug-that-looked-like-clean-code-3556)

**原文标题**: [The O(n^2) Bug That Looked Like Clean Code - DEV Community](https://dev.to/aralroca/the-on2-bug-that-looked-like-clean-code-3556)

本文揭示了看似整洁的代码中隐藏的O(n²)性能陷阱，并提供了识别和修复的方法。

- 🔍 核心问题：数组方法（如`.find()`、`.includes()`）嵌套在循环中导致O(n²)复杂度，在数据量增大时引发性能灾难
- 📉 典型案例：200用户时4万次比较，2000用户时400万次，20000用户时4亿次，导致API超时
- 🛠️ 修复方案：使用Map/Set替代数组查找，将O(n*m)降为O(n+m)，如用`new Map()`预建索引
- ⚠️ 五大常见模式：`.includes()`在`.filter()`内、用`findIndex`去重、链式方法中隐藏查找、递归树展开的数组复制、SQL查询循环（N+1问题）
- 💡 预防措施：用真实数据量测试、代码审查时检查嵌套查找、设置延迟告警、优先使用Map/Set
- 🎯 核心教训：性能复杂度是设计决策，不是后期优化；看似干净的代码可能成为定时炸弹

- 📚 **对用户影响小**：仅升级至LTS版本的用户几乎无影响，主要变化在版本编号；库作者需尽早集成Alpha版本到CI中。
- 🤝 **社区协作成果**：此变更经过GitHub讨论、工作组会议和协作峰会，未来将继续在伦敦峰会中探讨。
