### [React 19 错误边界行为差异 • 安德烈·卡拉赞斯](https://andrei-calazans.com/posts/react-19-error-boundary-changed/)

**原文标题**: [React 19 Error Boundary Behaves Differently • Andrei Calazans](https://andrei-calazans.com/posts/react-19-error-boundary-changed/)

React 19 中错误边界的行为发生了变化，现在遇到首个错误时会立即停止渲染，而不再继续处理同级组件的错误。

- 🚨 React 19 在错误边界中遇到首个错误时立即停止渲染，不再继续执行同级组件
- 🔄 React 18 会重复渲染错误组件并多次调用 componentDidCatch，造成重复错误日志
- ⚡ 新行为避免了资源浪费，因为首个错误发生后同级组件本就不会被渲染
- 🛠️ 这一改进使得开发者可以移除原有的错误日志去重逻辑
- 📝 示例显示只有第一个 Throws 组件被渲染，第二个组件完全不会执行

### [<Fragment>（<>...</>）– React](https://react.dev/reference/react/Fragment#using-fragment-refs-for-dom-interaction)

**原文标题**: [<Fragment> (<>...</>) – React](https://react.dev/reference/react/Fragment#using-fragment-refs-for-dom-interaction)

React Fragment（<Fragment>或<>...</>）用于在不添加额外 DOM 包装节点的情况下分组元素，支持引用（ref）以实现 DOM 交互、可见性追踪和焦点管理等功能。

- 🧩 用于分组元素而不产生额外 DOM 节点，等同于未分组状态
- 🔑 显式声明时可添加 key 属性，但短语法<>不支持
- 🎯 支持 ref 属性获取 Fragment 实例，用于 DOM 操作和事件处理
- 📋 适用于多元素返回、变量赋值、文本与组件混合等场景
- 🔄 循环渲染时需要显式声明 Fragment 并添加 key
- 👁️ 可通过 ref 实现可见性追踪和交叉观察
- ⌨️ 提供 focus()/focusLast() 等焦点管理方法
- ⚠️ 状态不会在<><Child/></>与<Child/>间重置
- 📝 短语法<>不支持 key 和 ref 属性，需显式导入 Fragment

### [JavaScript 源码映射的内部机制](https://www.polarsignals.com/blog/posts/2025/11/04/javascript-source-maps-internals)

**原文标题**: [The Inner Workings of JavaScript Source Maps](https://www.polarsignals.com/blog/posts/2025/11/04/javascript-source-maps-internals)

源映射通过 JSON 格式文件将压缩后的 JavaScript 代码位置映射回原始源代码位置，使开发者能在浏览器中直接调试原始代码。

- 🧩 源映射核心功能是将生产环境压缩代码（如 bundle.min.js:1:27698）反向映射到源码位置（如 src/index.ts:73:16）
- 🔄 构建流程三阶段均需源映射：转译（TypeScript→JavaScript）、打包（合并模块）、压缩（代码优化）
- 📜 源映射文件采用 JSON 格式，包含版本号、目标文件、源文件路径、标识符数组和关键映射数据
- 🗜️ 映射字段使用 VLQ 编码压缩技术，通过 Base64 字符存储位置差值而非绝对坐标
- 📍 映射段分三种类型：无源码映射（1 值）、基础位置映射（4 值）、含重命名标识符的映射（5 值）
- 🔢 VLQ 编码三步曲：符号位编码（最低位标识正负）、5 位数据分组（最高位作连续标志）、Base64 字符转换
- 🌰 数值 7 经 VLQ 编码后变为字符"O"，实现高效数据压缩
- 🚀 源映射技术即将扩展至性能分析领域，为性能剖析提供源码级调试支持

### [网页动画性能等级榜单 - 动感博客](https://motion.dev/blog/web-animation-performance-tier-list)

- 🚀 **S 级动画**：完全在合成线程运行，使用 transform/opacity/filter 等属性，即使主线程繁忙也能保持流畅
- ⚡ **A 级动画**：通过主线程驱动合成属性，需先通过 will-change 创建图层，性能优秀但可能被主线程阻塞
- 📐 **B 级动画**：使用 FLIP 技术进行前置 DOM 测量，随后转为 A/S 级动画，如 Motion 的布局动画引擎
- 🎨 **C 级动画**：触发绘制步骤，包括颜色变化、CSS 变量动画等，性能消耗随图层大小增加
- 📏 **D 级动画**：触发布局重排，导致整个渲染管线更新，成本取决于 DOM 树复杂度
- 💥 **F 级动画**：样式和布局的反复读写（抖动），会造成严重的性能问题
- 🎯 **优化要点**：优先使用合成线程动画，合理使用 will-change，避免 CSS 变量继承问题，通过 IntersectionObserver 管理可见区域动画
- ⚖️ **权衡因素**：GPU 内存占用、图层大小、模糊滤镜成本等都需要综合考虑
- 🔧 **工具推荐**：Motion 库提供批处理、延迟关键帧解析等功能，可有效避免性能问题
