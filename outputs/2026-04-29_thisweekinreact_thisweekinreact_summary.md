**原文标题**: [The React Compiler at Eighteen Months: The Arc, the Debates, and What's Next | Sascha Becker](https://saschb2b.com/blog/react-compiler-year-in-review)

### 概述总结
React Compiler 在发布十八个月后，已从初期炒作进入务实整合阶段，其最大贡献是消除了手动记忆化相关的错误类别，但生态系统的适配仍在进行中。

- 📅 **十八个月发展弧线**：从发布集成、安静的中期（减少重渲染错误）到生态清算（发现许多库违反 React 规则），目前绿场项目已解决，棕场项目仍需努力。
- ⚙️ **编译器核心功能**：构建时自动插入记忆化，消除`useMemo`/`useCallback`/`memo`，提升重渲染扩展性，但会增加构建时间和少量包体积。
- ❌ **编译器失效场景**：渲染期间修改 props/闭包、渲染中读取 ref、类组件、以及`"use no memo"`逃逸出口（需谨慎使用）。
- 🛠️ **推荐迁移路径**：先更新 React 版本 → 安装 ESLint 插件修复违规 → 注解模式试用 → 推理模式全面启用 → 批量删除手动记忆化代码。
- 🔥 **未解决的争议**：React 规则成为可执行契约（对旧库有冲击）、`"use no memo"`可能成为永久技术债、编译器与运行时优化器（如 Million.js）的互补关系。
- 🚀 **未来方向**：更细粒度的编译控制、编译器感知的服务器组件、`useEvent`收敛、React Native 优化、以及开发者工具可视化编译器行为。
- 💡 **作者观点**：编译器最大遗产是消除了"忘记 useCallback 依赖"这类错误，但生态系统的旧库仍需时间适配。新项目应直接启用，旧项目需权衡修复或使用逃逸出口。

### [React 如何无序流式输出 UI 却依然保持有序 | React 内部探秘](https://inside-react.vercel.app/blog/how-react-streams-ui-out-of-order)

**原文标题**: [How React streams UI out of order and still manages to keep order | Inside React](https://inside-react.vercel.app/blog/how-react-streams-ui-out-of-order)

React 通过 Suspense 边界实现无序流式渲染，同时保持 DOM 顺序的正确性。其核心机制是：先发送立即可用的 HTML，用模板占位符标记未就绪的组件，待数据就绪后通过 JavaScript 动态替换。

- 🧩 **核心问题**：传统 SSR 需等待所有数据加载完成才发送 HTML，导致页面阻塞；有序流式虽能逐步发送，但慢组件仍会阻塞后续内容。
- ⚡ **无序流式方案**：React 18 的`renderToPipeableStream`允许组件按数据就绪顺序独立发送，通过`<Suspense>`包裹慢组件并指定 fallback UI。
- 🏗️ **流式 HTML 结构**：服务器先发送`Navbar`、`Footer`及 Suspense 边界标记（`<!--$?-->`），每个边界包含`<template>`占位符和 fallback 内容。
- 🔄 **组件回传机制**：数据就绪后，服务器以隐藏`<div>`发送真实组件（如`<div hidden id="S:0">`），随后注入`<script>$RC("B:0","S:0")</script>`触发替换。
- 🛠️ **三大关键函数**：`$RC`负责匹配占位符和真实内容；`$RB`作为队列暂存待替换对；`$RV`在下一帧执行 DOM 替换，清除 fallback 并插入真实组件。
- 🔄 **Suspense 生命周期**：状态从`$?`（等待中）→`$~`（已就绪）→`$`（完成），通过注释节点数据变化追踪边界状态。
- ⚠️ **潜在风险**：若手动插入相同 ID 的`<template>`，React 会误替换该元素，导致 Suspense 边界失效。

本篇文章探讨了 React 应用中最常见的无障碍性问题，并提供了修复方法，涵盖语义化标签、焦点管理、元素标签和动态更新。

- 🔍 **使用正确的 HTML 元素**：用`<button>`替代`<div onClick>`，用`<a>`处理导航，利用浏览器原生功能，避免手动实现无障碍性。
- 📑 **维护文档结构与标题层级**：标题应按逻辑顺序（`<h1>`到`<h6>`），使用 CSS 控制样式而非标题级别；利用`<header>`、`<nav>`等地标元素帮助屏幕阅读器导航。
- 🏷️ **为所有交互元素添加标签**：表单输入需关联`<label>`，图标按钮使用`aria-label`或隐藏文本，图片添加`alt`属性（装饰性图片用`alt=""`）。

- 🔊 **宣布动态更新**：使用`role="alert"`（紧急）或`role="status"`（非紧急）让屏幕阅读器播报变化，保持容器在 DOM 中并更新内容。
- 🎨 **基于 ARIA 属性进行样式设计**：用`aria-selected`、`aria-disabled`等属性直接应用 CSS 样式，保持样式与无障碍状态一致。
- ✅ **快速审计清单**：检查`<div>`点击事件、标题层级、表单标签、图标按钮、图片 alt、表单错误、模态框焦点、条件渲染、动态更新，并手动 Tab 测试和运行自动化工具。

### [为什么 TSRX 不仅仅是您最喜欢的模板语言 - HackMD](https://hackmd.io/@0u1u3zEAQAO0iYWVAStEvw/Hk2Jiv8T-l)

TSRX 是一种新型模板语言，由 Dominic Gannaway 创建，目前已在 React、SolidJS 和 Ripple 中提供 Alpha 版本。它结合了 JavaScript 的灵活性和模板语言的静态分析能力，提供了独特的机械优势。

- 🎯 **核心优势**：TSRX 通过 `component` 函数作用域实现静态可分析的结构化，支持原生 JavaScript 条件/循环语句（如 `if`/`for`），而非表达式方式，这使其在编译器优化和 HMR 方面具有优势。
- 🔄 **状态化模板**：与需要重复渲染的 JSX 不同，TSRX 的模板无需反复执行，特别适合细粒度渲染的 UI 框架。通过将组件生命周期与状态解耦，状态可以安全地存在于任何位置。
- 📦 **可组合性**：TSRX 支持状态化模板的便携性，例如在条件语句中直接包裹代码块，无需像 JSX 那样修改多处代码。这降低了重构成本，并保持了代码的剪切粘贴能力。
- ⚖️ **与现有框架对比**：对于 Vue/Svelte 开发者，TSRX 提供了更接近 JavaScript 的语法；对于 React 开发者，它消除了 Hook 规则的限制并解锁了 Signals 库的潜力；对于 SolidJS 开发者，它可能成为理想的模板语言，但需权衡透明性。
- 🚀 **未来潜力**：TSRX 与 Mitosis 不同，它专注于提供用户愿意使用的语法，而非生成跨框架代码。在 AI 影响开发方式的时代，TSRX 的机械能力和设计理念可能开辟新空间。

### [智能体优先产品工程的黄金法则](https://newsletter.posthog.com/p/the-golden-rules-of-agent-first-product)

**原文标题**: [The golden rules of agent-first product engineering](https://newsletter.posthog.com/p/the-golden-rules-of-agent-first-product)

### 概述总结
代理优先的产品工程需将代理视为主要交互界面，而非附加功能，通过五大原则实现高效、可信赖的自主工作流。

- 🎯 **原则一：赋予代理与用户同等能力**  
  确保代理能执行用户所有操作（如 A/B 测试创建），避免因工具缺失被迫中断流程。实践中通过 OpenAPI 自动生成工具处理器，覆盖所有 API 端点。

- 🧩 **原则二：匹配代理的抽象层级**  
  使用代理“原生语言”（如 SQL）而非人类 UI 术语，减少上下文消耗。例如用单条 SQL 查询替代多次 API 调用，提升效率并释放创造力。

- 📦 **原则三：前置通用上下文**  
  在会话开始时加载产品特定知识（如术语、SQL 语法、硬性约束），避免代理重复探索。如 PostHog MCP 中预置事件分类和查询规则。

- 📝 **原则四：技能文档聚焦隐性知识**  
  技能文档应包含代理无法自行发现的内部规则（如默认使用`$pageview`事件），而非步骤说明书。通过注入领域专长，确保代理输出符合产品最佳实践。

- 👤 **原则五：将代理视为真实用户**  
  通过无头模式测试、人工追踪审查、构建评估用例等方式，理解代理行为模式。例如每周分析用户会话，将异常案例转化为自动化评估标准。

本 RFC 提出了在 React Native iOS 项目中用 Swift Package Manager（SPM）取代 CocoaPods 的计划，并提供了技术实施方案。

- 📜 **RFC 核心目标**：在 React Native iOS 项目中逐步淘汰 CocoaPods，全面采用 Swift Package Manager，并已获得多位核心维护者批准。
- ⚙️ **自动化集成流程**：通过`setup-ios-spm.js`脚本自动完成代码生成、XCFramework 下载、Package.swift 文件创建及 Xcode 项目配置，简化开发者操作。

- 🛠️ **构建与调试限制**：第一阶段不支持从源码构建 React Native，调试时需回退到 CocoaPods；后续版本将支持 SPM 插件实现预构建脚本。
- 🏢 **棕地项目适配**：支持通过独立 Package.swift 文件生成 XCFramework 或源码包，便于集成到现有原生应用中，但需解决框架 target 的依赖继承问题。
- 🔄 **静态/动态框架切换**：SPM 通过 Package.swift 引用预构建 XCFramework 或源码，无需类似 CocoaPods 的环境变量切换方式。
- 📦 **第三方库迁移路径**：长期方向是在`react-native.config.js`中添加 SPM 元数据，保持依赖发现统一，由 CLI 决定集成方式。
- 🚧 **已知限制与待办**：需解决 SPM 插件自定义脚本钩子、构建阶段脚本支持、以及棕地场景下的框架分发等问题。

### [第一部分：如何在 React Native 中让纯 JSI 代码运行更快](https://blog.margelo.com/make-jsi-run-faster)

**原文标题**: [Part 1: How to Make Pure JSI Code Faster in React Native](https://blog.margelo.com/make-jsi-run-faster)

### 概述总结
JSI 代码本身很快，但架构选择可导致 2-10 倍性能差异。优化关键在于减少 JS 与 C++ 边界跨越、属性解析开销及内存分配。

- ⚡ **HostFunction 比 HostObject 快 5 倍**：直接使用 HostFunction 可避免每次调用时虚拟调度、字符串转换和属性查找的开销。
- 🧩 **NativeState 比 HostObject 快 5 倍**：通过`Object + NativeState`替代 HostObject，消除动态属性解析层，直接访问原生指针。
- 📦 **栈分配比堆分配快 3 倍**：已知大小（如<512 字节）时用`char buf[256]`替代`std::string`，避免堆分配和 SSO 限制。
- 🔤 **选择正确的字符串构造函数**：ASCII 用`createFromAscii`、UTF-8/16 用对应方法，避免不必要的编码转换和 null 终止符扫描。
- ⏱ **减少 JS↔C++ 边界跨越**：合并多次小调用为一次大调用，避免每次调用的参数转换和返回值包装开销。
- 🏎 **哈希查找优化属性分发**：固定属性集时用哈希替代字符串比较链，减少分支和重复字符串操作。
- 🛠 **Nitro Modules 自动优化**：内置 PropNameID 缓存、哈希分发和 NativeState 支持，无需手动实现底层优化。
- 🎯 **关注热路径**：非频繁执行的代码无需过度优化，但热路径上的微小选择可带来显著性能提升。

本集探讨了如何编写可维护的 CSS，重点在于防止样式泄露、采用流式布局，并选择合适的方法论（如工具类 CSS、组件作用域样式或 CSS 模块），同时利用 CSS 变量、层、作用域和工具来保持样式表整洁可扩展。

- 🎯 **防止样式泄露**：CSS 不应影响网站其他部分，确保样式隔离。
- 🛠️ **易于维护**：代码应简单明了，方便后续修改和更新。
- ♻️ **可重用性**：设计可复用的样式，减少重复代码。
- 🌍 **全局解决方案优先**：用全局方法解决常见问题，而非局部修补。
- 🔄 **灵活性与适应性**：CSS 应能适应不同屏幕和场景。
- 📏 **流式排版与响应式设计**：使用流体排版（如`fluid-type`）实现自适应布局。
- 🎨 **变量与一致性**：通过 CSS 变量保持设计一致性，区分值与变量。
- 📚 **选择正确方法论**：根据项目需求选择工具类 CSS（如 StyleX、Panda CSS）、组件作用域 CSS 或 CSS 模块。
- 🧩 **组件作用域 CSS**：推荐使用组件级样式，避免全局冲突。
- 📦 **CSS 模块演进**：利用现代 CSS 模块功能提升可维护性。
- 🌐 **全局 CSS 最佳实践**：如 Josh Comeau 的 CSS Reset 和 Scott 的 Graffiti UI，优化全局样式。
- 🎛️ **CSS 变量应用**：善用变量管理主题和动态样式。
- 🧱 **CSS 层（Layers）**：通过层控制样式优先级，减少冲突。
- 🔒 **CSS 作用域（Scoping）**：使用`@scope`等特性限制样式影响范围。
- ⚙️ **工具强制执行标准**：借助工具（如 Sentry、Project Wallace）确保 CSS 规范。
