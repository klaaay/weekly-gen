### [在 Next.js 16 中实现 next-intl 国际化缓存功能 | Aurora Scharff](https://aurorascharff.no/posts/implementing-nextjs-16-use-cache-with-next-intl-internationalization/)

### [Next.js 错误处理与 catchError | Aurora Scharff](https://aurorascharff.no/posts/error-handling-in-nextjs-with-catch-error)

- 🚫 `react-error-boundary`无法区分普通错误与 Next.js 控制流错误（如`notFound()`、`redirect()`），导致意外捕获框架错误。
- 🔄 该库的错误恢复机制`resetErrorBoundary`仅清除客户端状态，不会重新获取服务端数据，导致重试无效。
- 🛠️ 临时解决方案需手动检测错误摘要并配合`router.refresh()`与组件`key`重置来实现正确恢复。
- 🆕 `unstable_catchError`是框架感知的错误边界，能自动放行 Next.js 控制流错误，并提供`retry()`函数来重新获取服务端数据。
- 📄 同样的`retry()`模式也适用于路由层级的`error.tsx`文件，用于重新渲染整个路由段。
- ✅ 推荐使用`unstable_catchError`替代自定义方案，以简化代码并确保与 Next.js 框架行为一致。

### [后 React 编译器 React 编码指南（面向 AI 代理）](https://pavi2410.com/blog/post-react-compiler-coding-guide/)

**原文标题**: [Post-React Compiler React Coding Guide (For AI Agents)](https://pavi2410.com/blog/post-react-compiler-coding-guide/)

本文作者针对 AI 代码生成工具编写 React 指南，旨在帮助 AI 跟上 React 19 和 React Compiler v1.0 的最新变化。当前 AI 生成的 React 代码仍在使用过时的模式，如过度使用`useCallback`、`useMemo`和`forwardRef`，而未能利用新特性如 ref 回调的清理函数和编译器的自动优化。本指南是系列文章的第一篇，重点介绍“后 React 编译器时代”的编码模式，强调编写简洁、可读、声明式且编译器友好的代码，避免不必要的性能优化模式。核心思想是：开发者应专注于代码的清晰与正确性，将性能优化交给编译器处理。

- 🧠 **思维模式转变**：从手动控制渲染（使用`useMemo`、`useCallback`）转向信任编译器自动优化，编写代码时假设每次渲染都是免费的。
- ⚙️ **核心原则**：优先使用纯 JavaScript（变量、函数），避免默认使用手动记忆化；将组件视为纯函数，仅根据 props、state 和 context 推导 UI。
- 🪝 **Hooks 使用指南**：`useState`用于真正的本地 UI 状态；`useEffect`仅用于副作用（如同步外部系统），而非数据推导；`useRef`用于不触发渲染的可变值或命令式句柄。
- 🔄 **Props 与数据流**：内联回调函数是可接受的（编译器会优化）；仅当语义需要时才提升状态（例如多个组件需要协调）。
- 📋 **列表与键**：使用稳定且有语义的键（如`item.id`），避免使用索引；不要手动记忆化列表渲染。
- 🧮 **派生数据**：在渲染中直接计算派生数据，而非将其存储在 state 中。
- 🚨 **错误边界与 Suspense**：将`Suspense`视为控制流原语（如加载状态），而非性能技巧。
- ⚡ **手动优化的适用场景**：仅在性能分析显示瓶颈、与非 React 系统交互或需要引用稳定性以保证正确性时，才使用`useMemo`等作为“逃生舱口”，并需文档说明原因。
- ⚙️ **编译器配置与指令**：AI 应假设默认使用`infer`模式，遵循命名约定（如 PascalCase 组件、`use*`前缀的 Hooks）以辅助编译器推断；谨慎使用指令（如`"use memo"`、`"use no memo"`），仅当必要时才引入。
- ✨ **代码风格期望**：AI 生成的代码应注重可读性，避免防御性模式，最小化 Hooks 使用，除非必要否则不添加关于记忆化的注释。
- ✅ **总结规则**：不要默认使用`useMemo`/`useCallback`；不存储派生状态；不手动优化渲染；编写纯函数组件；内联推导值；仅基于语义使用 Hooks；信任编译器。

### [在 CI 中使用 maxWorkers 修复不稳定的 Jest 测试（及其原理） | 作者：Tyler | 2026 年 3 月 | Red Shift](https://shift.infinite.red/fix-flaky-jest-tests-in-ci-with-maxworkers-and-why-it-works-e3d3189f35a4?gi=cdb313f4189c)

**原文标题**: [Fix Flaky Jest Tests in CI with maxWorkers (and why it works) | by Tyler | Mar, 2026 | Red Shift](https://shift.infinite.red/fix-flaky-jest-tests-in-ci-with-maxworkers-and-why-it-works-e3d3189f35a4?gi=cdb313f4189c)

Jest 单元测试在 CI/CD 中可能因内存压力导致不稳定，通过限制并行工作线程数可有效缓解此问题。

- 🧪 **Jest 默认并行化过度消耗内存**：Jest 会尽可能多地占用内存，导致系统内存不足，进而引发非确定性减速和测试超时失败。
- ⏱️ **超时断言在内存压力下易失效**：如示例中`setTimeout`延迟 4 秒的组件，在内存不足时可能超过测试设置的 5 秒超时，造成间歇性失败。
- 🔧 **解决方案：使用`--maxWorkers`限制工作线程**：通过设置`--maxWorkers='50%'`（或具体数值）减少并行线程，降低内存压力，提升测试稳定性。

- 📊 **无交换内存环境下同样有效**：即使在没有交换内存的 CI 机器上，内存抖动也会导致减速，`maxWorkers`能通过减少线程数避免过度内存竞争。
