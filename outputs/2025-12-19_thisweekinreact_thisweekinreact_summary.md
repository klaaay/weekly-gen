- 🔓 **中危源代码泄露漏洞**：攻击者可能通过特定请求获取服务器函数的源代码，可能暴露硬编码的敏感信息

- 📦 **受影响包**：`react-server-dom-webpack`、`react-server-dom-parcel`、`react-server-dom-turbopack`及依赖它们的框架（如 Next.js、React Router）
- 🛡️ **临时缓解措施**：部分托管服务商已实施临时防护，但不应依赖此替代正式更新
- 📅 **漏洞时间线**：12 月初由安全研究人员报告，12 月 11 日发布补丁并分配 CVE 编号
- ℹ️ **适用范围说明**：仅使用 React Server Components 的应用程序受影响，纯客户端应用不受影响

### [安全通告](https://storybook.js.org/blog/security-advisory/)

**原文标题**: [Security advisory](https://storybook.js.org/blog/security-advisory/)

Storybook 团队于 2025 年 12 月 11 日收到关于 Storybook 7 至 10 版本中存在安全漏洞的负责任披露。该漏洞源于 Storybook 在处理.env 文件中的环境变量时存在缺陷，可能导致敏感信息被意外打包进构建产物中，若将构建后的 Storybook 发布到网上，这些信息可能被公开访问。团队已发布修复版本，并建议受影响用户立即升级、审计并轮换敏感密钥。

- 🔒 **漏洞概述**：Storybook 7.0.0 及以上版本在构建时，若项目目录中存在.env 文件且代码中使用了特定的`process.env`引用模式，可能导致文件中的所有环境变量被捆绑到公开的构建产物中，造成敏感信息泄露。
- 📁 **影响条件**：项目需同时满足使用受影响版本、在包含.env 文件的目录中构建 Storybook、文件中包含敏感密钥，并将构建结果发布到网上。本地开发环境或仅通过平台环境变量传递密钥的 CI 构建不受影响。
- 🛠️ **修复措施**：立即升级 Storybook 至安全版本（10.1.10+、9.1.17+、8.6.15+ 或 7.6.21+），审计并轮换可能已泄露的密钥。若升级后环境变量读取异常，可改用`STORYBOOK_`前缀或在配置中手动设置值，但切勿包含敏感信息。
- ⚠️ **触发模式**：当`manager.js|ts`或插件代码中以对象形式动态引用`process.env`（如`{ ...process.env }`）或引用未明确配置的变量时，会触发漏洞。直接访问已知属性（如`process.env.MY_VAR`）则安全。
- 📦 **受影响插件**：审计发现`@chromatic-com/storybook`、`@storybook/addon-designs`和`@amplitude/storybook-addon-amplitude`等常用插件可能触发漏洞，但漏洞根源在 Storybook 本身，插件无需单独修改。
- 🔐 **额外建议**：使用 Chromatic 发布的项目已采取缓解措施，但仍建议遵循通用修复步骤。团队感谢报告者 Matt G，并鼓励用户订阅邮件列表获取更新。

### [React 服务器组件性能简介 - 网页性能日历](https://calendar.perfplanet.com/2025/intro-to-performance-of-react-server-components/)

**原文标题**: [  Intro to Performance of React Server Components - Web Performance Calendar](https://calendar.perfplanet.com/2025/intro-to-performance-of-react-server-components/)

本文探讨了 React Server Components（RSC）在性能方面的表现，通过对比客户端渲染（CSR）、服务端渲染（SSR）及 RSC 的实验数据，分析了它们对初始加载时间（LCP）、动态内容可见性和交互延迟的影响。文章指出，RSC 在数据获取场景下能显著提升性能，但实现复杂且存在技术风险，需权衡其收益与成本。

- 🚀 **RSC 的核心优势**：通过将更多工作移至服务器、减少 JavaScript 传输和提前获取数据，优化初始加载性能。
- ⏱️ **实验设置**：构建单页应用模拟不同渲染方式，测量 LCP、侧边栏和消息列表的可见时间。
- 📊 **CSR 表现**：初始加载慢（LCP 约 4.1 秒），用户需等待 JavaScript 下载执行后才能看到界面。
- 🌐 **SSR 改进**：服务端预渲染 HTML 使 LCP 大幅提升至 1.61 秒，但动态数据仍依赖客户端获取，且存在约 2.39 秒的“无交互期”。
- 🔄 **SSR 结合数据获取**：在服务端获取数据可加速动态内容显示（如消息列表从 5.1 秒减至 1.78 秒），但可能延长 LCP 时间。
- ⚡ **RSC 与流式渲染**：支持渐进式渲染，实现更快的 LCP（1.28 秒）和动态内容加载，但需保持长连接，可能增加服务器负载。
- ⚠️ **实施挑战**：RSC 需全面重构应用架构，技术尚未成熟，存在学习曲线、工具链限制和安全风险。
- 🤔 **适用性建议**：仅在数据获取关键的场景中考虑 RSC；若仅关注静态内容或交互性，传统 SSR 可能更简单有效。

- 🖼️ **窗口化与布局行为**：VR 环境中的应用窗口可被用户自由移动和调整大小，因此响应式设计成为首要需求，需使用 Flexbox 等布局系统适应动态视口。
- 🎨 **样式设计**：VR 中的文本、图标和颜色需特别注重可读性和舒适性，建议使用更大的字体、简洁的图标和高对比度色彩，避免视觉疲劳。
- ♿ **可访问性与舒适性**：设计需考虑控制器交互的简便性、减少头部移动、提供充足的反馈（如音效和触觉反馈），并适配不同交互方式（控制器或手部追踪）。
- 🖱️ **交互设计**：VR 交互强调即时反馈，需利用悬停状态和动画增强用户体验，确保触摸目标足够大（至少 48dp×48dp），并使用相关库实现流畅交互。
- 📱 **平台特定调整**：可通过检测设备平台（如使用 ExpoHorizon.isHorizonDevice）来微调布局和样式，同时需注意 VR 中缺少系统返回键，应提供明确的应用内返回控制。
- 🤖 **辅助工具**：可利用 AI 工具和 VRC（虚拟现实检查）来审查代码是否符合 Meta Quest 的设计要求，确保应用质量。

- 🔧 **死代码消除**：Hermes 编译器在生成字节码时自动移除未使用的代码分支，无需额外压缩工具。
- 📦 **构建效率**：关闭压缩可加速构建过程，且代码体积差异极小（如示例中仅差 0.17MB）。
- 🔍 **验证方法**：可通过 `hermes-dec` 工具反编译字节码文件，检查生产包中死代码是否已被清除。
- 💡 **开发建议**：理解 React Native 工具链的优化机制，避免不必要的操作，专注于 Hermes 带来的性能提升。

### [为什么我的视图转换会闪烁？- Piccalilli](https://piccalil.li/blog/why-are-my-view-transitions-blinking/)

**原文标题**: [
  Why are my view transitions blinking? - Piccalilli
](https://piccalil.li/blog/why-are-my-view-transitions-blinking/)

视图过渡（View Transitions）API 可实现平滑的页面状态切换动画，但若使用不当，会出现闪烁问题。闪烁通常是由于浏览器无法在旧状态和新状态之间正确关联元素导致的。

- 🚨 **闪烁原因**：当元素在过渡前后无法通过 `view-transition-name` 正确关联时，浏览器会回退到简单的交叉淡入淡出效果，导致内容突然消失和重现。
- 🔍 **常见错误**：旧元素未设置 `view-transition-name`、多个元素共享相同名称、使用 CSS 隐藏而非移除 DOM 元素，或名称设置时机不当。
- 💡 **解决方案**：在调用 `startViewTransition()` 前为旧元素设置唯一 `view-transition-name`，过渡中创建新元素并赋予相同名称，过渡完成后清理名称。
- 🛠️ **正确实践**：通过真正的 DOM 操作（如移除旧元素、添加新元素）而非仅切换可见性，确保元素在过渡前后有清晰的状态标识。
- 📋 **实施清单**：过渡前标识并命名旧元素；过渡中执行最小 DOM 操作并命名新元素；过渡后清理名称并维护可访问性。
- ⚠️ **注意事项**：避免重复的 `view-transition-name`、注意设置时机、清理残留名称，并为不支持该 API 的浏览器提供回退方案。
