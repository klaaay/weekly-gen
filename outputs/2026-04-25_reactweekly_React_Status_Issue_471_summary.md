### [现在你看到了：无需代理的Vite on Rails——火星编年史，Evil Martians团队博客](https://evilmartians.com/chronicles/now-you-see-it-vite-on-rails-without-the-proxy)

**原文标题**: [Now you see it: Vite on Rails without the proxy—Martian Chronicles, Evil Martians’ team blog](https://evilmartians.com/chronicles/now-you-see-it-vite-on-rails-without-the-proxy)

## 概述总结

Vite 与 Rails 的 Propshaft 资产管道原本不兼容，但通过创新的“存根文件”技巧，实现了无需代理的无缝集成，让开发者既能享受 Vite 的热模块替换等现代功能，又能保持 Rails 原有工作流。

- 🔥 **Vite 的核心优势**：提供热模块替换（HMR）、即时 CSS 反馈、单进程处理 JS/CSS/Tailwind、自动代码分割，以及丰富的插件生态（React、Vue、Svelte 等）
- 🧩 **解决兼容性的巧妙方法**：在 `app/assets/builds/` 中创建存根文件，通过 import 语句将浏览器重定向到 Vite 开发服务器，Propshaft 认为提供了真实文件，浏览器则直接与 Vite 通信
- ⚡ **CSS 处理技巧**：CSS 通过 JS import 注入 `<style>` 标签实现 HMR，开发时 `stylesheet_link_tag` 指向空文件避免错误，生产环境则生成真实 CSS 文件
- 🚀 **两种集成模式**：jsbundling 模式（存根文件+生产双文件，适合简单迁移）和 gem 模式（通过 `vite_tags` 直接输出标签，支持 CSP nonce、`vite_image_tag` 等高级功能）
- 🔄 **从 esbuild 迁移示例**：仅需三个步骤——安装 npm 包、添加 Vite 配置文件、更新 Procfile，即可替换原有构建工具
- 🛠 **部署无变化**：`yarn build` 替换为 `vite build`，CI/Docker/Kamal/Heroku 等部署流程完全不变
- ⚠️ **注意事项**：`package.json` 需添加 `"type": "module"`，PostCSS/Tailwind 配置文件需重命名为 `.cjs`，Stimulus 控制器需改用 `import.meta.glob`
- 🎩 **效果总结**：Vite 对 Rails 透明不可见，两者各取所需无需妥协，已在 Evil Martians 客户项目和生产环境中验证
