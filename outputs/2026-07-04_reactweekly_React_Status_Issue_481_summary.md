**原文标题**: [Chasing a Phantom Jump: How We Made Pingo's Animations Smooth on Low-End Android — Margelo Blog](https://blog.margelo.com/profiling-skia-reanimated-low-end-android)

### 概述总结
本文详细记录了在廉价120Hz安卓设备上修复Pingo应用动画卡顿问题的完整过程，揭示了动画数学正确但用户感知卡顿的根本原因在于帧交付时序而非动画曲线本身。

- 🎯 **核心发现**：动画平滑度取决于帧节奏而非位置曲线，数学上完美的弹簧动画因帧交付延迟导致用户感知卡顿
- 🔍 **主要瓶颈**：Skia在Android上的TextureView合成方式导致每帧需进行2-3次GPU纹理上传，在廉价120Hz设备上远超8.3ms预算
- 📊 **诊断方法**：通过`dumpsys gfxinfo`发现85%卡顿帧，降低刷新率至60Hz后卡顿降至15%，确认问题为固定GPU开销超出帧预算
- 🛠️ **最大修复**：将`<Canvas opaque>`从TextureView切换为SurfaceView，使画布拥有独立SurfaceFlinger层，消除每帧纹理上传，RenderThread CPU降低50-65%
- 🔄 **次要问题**：`usePathValue`/`usePathInterpolation`钩子存在自循环重绘，通过缓冲区突变模式修复，空闲画布帧时间从17-32ms降至~9ms
- ⚡ **React优化**：消除因Zustand状态对象引用变化导致的非必要重渲染，以及通过工作集驱动状态而非React提交
- 🤖 **自动化调试**：使用LLM代理并行自动化构建-导航-测量循环，加速瓶颈定位，但假设判断仍需人工完成
- 📈 **最终效果**：在相同廉价设备上，动画从85%卡顿帧变为流畅运行，用户不再感知跳帧

### [仪表化 | React Router](https://reactrouter.com/how-to/instrumentation#result-metadata)

**原文标题**: [Instrumentation  | React Router](https://reactrouter.com/how-to/instrumentation#result-metadata)

React Router 的 Instrumentation API 允许你在不修改实际路由处理程序的情况下，添加日志记录、错误报告和性能追踪，实现全面的可观测性。

- 📝 **概述**：提供“包装器”函数，在请求处理程序、路由器操作、路由中间件等周围执行，用于监控性能、添加日志、集成可观测性平台等。
- 🚀 **框架模式快速开始**：在 `entry.server.tsx` 和 `entry.client.tsx` 中添加 `instrumentations` 数组，分别对服务器端和客户端进行插桩。
- 💻 **数据模式快速开始**：在创建路由器时（如 `createBrowserRouter`）传入 `instrumentations` 配置。
- 🎯 **核心概念**：支持处理程序级（服务器）、路由器级（客户端）和路由级（服务器+客户端）的插桩，每个级别提供不同的上下文信息。
- 🔒 **只读设计**：插桩仅用于观察，不能修改参数、返回值或改变应用行为，确保生产环境安全。
- 🛡️ **错误处理**：插桩代码中的错误被内部捕获，不会影响应用或其他插桩函数；处理程序错误通过返回 `{ status, error }` 对象来传递。
- 📊 **结果元数据**：客户端导航/获取和服务器请求处理程序返回 `meta` 字段，包含 `url`、`pattern`、`params` 和 `statusCode`。
- 🧩 **组合与条件插桩**：可以组合多个插桩形成执行链，并可根据环境或条件启用/禁用插桩。
- ✨ **常见模式**：包括服务器请求日志记录、OpenTelemetry 集成和客户端性能追踪的示例代码。
