- 🚀 **性能飞跃**：TypeScript 7.0 将代码库从 TypeScript 移植到 Go，利用原生代码速度和共享内存并行，速度比 6.0 快约 10 倍。

## 概述总结

Vite 为 Rails 开发者带来了热模块替换、即时 CSS 反馈、代码分割等现代前端体验，而 `rails_vite` 插件通过巧妙的“存根文件”机制，让 Vite 与 Propshaft 资产管道无需代理即可无缝协作，既保持了 Rails 惯例，又无需大幅改动现有工作流。

- 🚀 **Vite 的核心优势**：热模块替换（HMR）让 React/Vue/Stimulus 组件更新不丢失状态；CSS 修改即时生效，无需页面刷新；单个开发服务器替代多个 watcher；自动代码分割提升加载性能；丰富的插件生态与 Rust 构建工具 Rolldown 带来极速构建。

- 🧩 **存根文件（Stub）机制**：在 `app/assets/builds/` 中放置一个微型脚本，将浏览器从 Propshaft 重定向到 Vite 开发服务器。Propshaft 以为提供了真实文件，浏览器实际连接 Vite，双方无需妥协。

- 🎨 **CSS 处理技巧**：通过 JavaScript 导入 CSS 文件，Vite 返回注入 `<style>` 标签的 JS 模块，实现 CSS HMR 无需页面刷新。生产环境则生成真实 CSS 文件通过 `<link>` 标签加载。

- 🔄 **生产环境双文件（Double）机制**：Vite 输出内容哈希后的文件到 `public/assets/`，插件在 `app/assets/builds/` 写入重导出哈希包的双文件，确保模块身份一致，浏览器正确加载。

- 📦 **迁移简单**：从 esbuild 迁移只需三步：安装 npm 包、添加 Vite 配置、更新 Procfile。Tailwind 用户可替换为 `@tailwindcss/vite`，移除独立 CSS watcher。

- ⚠️ **注意事项**：`package.json` 添加 `"type": "module"` 后，`postcss.config.js` 需改为 `.cjs` 扩展名；Stimulus 控制器需改用 `import.meta.glob` 配合 `stimulus-vite-helpers`。

- 🚢 **部署无变化**：修改 `package.json` 的构建脚本为 `"vite build"`，CI/Docker/Kamal 等设置无需改动，`jsbundling-rails` 自动处理 `assets:precompile`。

- 💎 **Gem 模式（高级选项）**：通过 `rails_vite` gem 和 `vite_tags` 助手，浏览器直接连接 Vite，无存根层，支持 `vite_image_tag`、CSP nonce、`modulepreload` 提示，适合需要更精细控制的场景。

- 🎩 **魔术揭秘**：`rails_vite` 让 Vite 在 Rails 中“隐形”，两种模式（jsbundling 存根模式与 gem 直接模式）均无需代理，双方各取所需，无需妥协。

### [NPM 安全 - OWASP 速查表系列](https://cheatsheetseries.owasp.org/cheatsheets/NPM_Security_Cheat_Sheet.html)

**原文标题**: [NPM Security - OWASP Cheat Sheet Series](https://cheatsheetseries.owasp.org/cheatsheets/NPM_Security_Cheat_Sheet.html)

本速查表提供了全面的npm安全最佳实践，涵盖从发布前检查到供应链保护等关键环节，旨在帮助JavaScript和Node.js开发者有效防范安全风险。

- 📜 **避免向npm注册表泄露密钥**：使用`.npmignore`或`files`属性控制发布内容，并用`--dry-run`预览tarball，防止密钥意外泄露。
- 🔒 **强制使用锁文件**：通过`npm ci`或`yarn install --frozen-lockfile`确保依赖版本一致性，避免因锁文件不同步引入意外变更。
- 🛡️ **通过忽略运行脚本减少攻击面**：使用`--ignore-scripts`禁用第三方包的脚本执行，并借助`@lavamoat/allow-scripts`创建白名单，仅允许可信包运行生命周期脚本。
- 🔍 **评估npm项目健康度**：利用`npm outdated`检查依赖更新状态，用`npm doctor`诊断环境配置，确保npm安装和工作环境正常。
- 🕵️ **审计开源依赖漏洞**：持续扫描第三方开源项目中的安全漏洞，并在整个开发生命周期内监控，及时接收新CVE影响的通知。
- 📦 **制品治理与供应链保护**：使用本地npm代理（如Verdaccio）管理依赖，生成SBOM（如CycloneDX），签名制品并验证完整性，限制并轮换CI/发布令牌，自动化监控异常行为。
- 📢 **负责任地披露安全漏洞**：遵循负责任披露流程，与维护者协调修复和发布日期，确保用户有升级路径后再公开漏洞信息。
- 🔐 **启用双因素认证（2FA）**：通过npm CLI或网站启用2FA，保护账户安全，防止未授权访问和操作。
- 🗝️ **使用npm作者令牌**：创建只读或IP范围限制的令牌用于CI和自动化流程，并定期检查与撤销令牌，减少令牌暴露风险。
- 🎭 **理解打字错误和懒惰攻击**：警惕typosquatting（利用拼写错误）和slopsquatting（利用AI幻觉）攻击，安装前用`npm view`验证包存在性、下载量和仓库真实性。
- ✅ **使用可信发布者安全发布包**：通过OIDC实现可信发布，使用短期工作流凭证替代长期令牌，并自动生成来源证明，增强发布安全性。
- 🚫 **防止依赖混淆攻击**：为内部包使用作用域名（如`@yourorg/package-name`），配置`.npmrc`指向私有注册表，并在公共注册表预留名称占位。
- 📝 **验证文档示例后再复制到生产环境**：警惕文档示例中的不安全模式（如弱密钥派生、未锚定正则），复制前进行安全审查，优先采用库的内部安全默认值。
- 💡 **最终建议**：谨慎复制安装命令，日常使用npm登出状态，安装时附加`--ignore-scripts`，以降低账户泄露和任意命令执行风险。
