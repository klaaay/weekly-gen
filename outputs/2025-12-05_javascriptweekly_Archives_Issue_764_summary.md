### [1995 年，一位网景员工用 10 天写出的代码如今支撑着整个互联网——Ars Technica](https://arstechnica.com/gadgets/2025/12/in-1995-a-netscape-employee-wrote-a-hack-in-10-days-that-now-runs-the-internet/)

**原文标题**: [In 1995, a Netscape employee wrote a hack in 10 days that now runs the Internet - Ars Technica](https://arstechnica.com/gadgets/2025/12/in-1995-a-netscape-employee-wrote-a-hack-in-10-days-that-now-runs-the-internet/)

JavaScript 诞生于 1995 年，由 Netscape 工程师 Brendan Eich 在十天内创建原型，现已成为支撑全球近 99% 网站客户端代码的核心编程语言，并广泛应用于服务器、移动应用和桌面软件等领域。

- 🚀 **诞生传奇**：1995 年 5 月，Brendan Eich 在 Netscape 公司用 10 天时间快速开发出 JavaScript 的原型，最初命名为“Mocha”，后改为“LiveScript”，最终为借助 Java 热度定名为“JavaScript”。
- 🌐 **广泛影响**：JavaScript 目前运行于约 98.9% 的网站客户端，是世界上最流行的编程语言之一，并借助 Node.js 等技术扩展到服务器、移动应用和桌面软件开发。
- 🔄 **命名与混淆**：语言名称是为营销而取，与 Java 几乎无关，两者在设计和用途上差异显著，常被比喻为“汽车”和“地毯”的关系。
- 📜 **标准化历程**：1997 年通过 ECMA International 标准化为 ECMAScript；早期因浏览器兼容性问题发展坎坷，2005 年 AJAX 技术兴起后重获活力。
- 🛠️ **生态扩张**：如今 npm 托管约 200 万至 300 万个包，TypeScript 作为 JavaScript 超集迅速增长，框架如 React Native 和 Electron 进一步拓展其应用范围。
- ⚖️ **商标争议**：Oracle 持有 JavaScript 商标但未使用，社区于 2024 年发起请愿要求取消商标，以促进“JavaScript”名称的自由使用。
- 🎉 **持久遗产**：尽管设计存在缺陷且初衷为辅助 Java，JavaScript 却超越众多早期科技公司，成为 Web 开发不可或缺的基石。

### [JavaScript™](https://javascript.tm/letter)

**原文标题**: [JavaScript™](https://javascript.tm/letter)

- 🏷️ Oracle 持有 JavaScript 商标，但从未推出名为 JavaScript 的产品，商标仅为其收购 Sun Microsystems 的遗留物。
- ⚖️ 根据美国法律，商标若连续三年未使用或成为通用术语即视为放弃；JavaScript 符合这两项条件。
- 🔄 JavaScript 已发展为独立编程语言，被全球开发者广泛使用，与 Oracle 无关，商标所有权与实际使用严重脱节。
- 📜 Oracle 在商标使用证据中引用非其产品（如 Node.js）及内部库（JET），但缺乏真实商业使用，无法满足商标法要求。
- 🌍 术语“JavaScript”已成为通用名称，但商标限制导致社区活动（如会议）无法直接使用该名称，造成混乱与不便。
- ✍️ 文章呼吁 Oracle 主动释放商标至公共领域，否则将向美国专利商标局提交撤销请愿，并征集公众签名及法律支持。

### [TypeScript 7 进展 - 2025 年 12 月 - TypeScript](https://devblogs.microsoft.com/typescript/progress-on-typescript-7-december-2025/)

**原文标题**: [Progress on TypeScript 7 - December 2025 - TypeScript](https://devblogs.microsoft.com/typescript/progress-on-typescript-7-december-2025/)

TypeScript 7.0（代号“Corsa”）作为原生代码重写版本，在编译器和语言服务方面取得重大进展，已提供稳定可用的预览版，性能显著提升，并计划作为 TypeScript 6.0 之后的下一个主要版本。

- 🚀 **编辑器支持完善**：VS Code 扩展预览版已支持代码补全（含自动导入）、跳转定义、重命名等核心功能，日常使用稳定。
- ⚡ **编译性能大幅提升**：原生编译器`tsgo`相比 TypeScript 6.0 在全量构建中可达近 10 倍速度提升，并支持增量编译与多项目并行构建。
- ✅ **类型检查高度兼容**：在数千个测试用例中，TypeScript 7.0 与 6.0 的错误检测一致性极高，可安全用于项目类型验证。
- ⚠️ **注意行为变更**：将默认启用`--strict`、移除`--target es5`等已弃用功能，部分 JavaScript 类型检查规则更严格（如不再支持`@enum`标签）。
- 🔄 **过渡策略明确**：TypeScript 6.0 将是最后一个基于 JavaScript 代码库的版本，后续主要开发将集中于 7.0，两者可并行使用以实现平稳迁移。
- 🛠️ **功能限制说明**：目前 JavaScript 发射管道仅支持到`es2021`目标，暂不支持装饰器编译，且新的 API 尚未稳定，第三方工具集成需等待。
- 📢 **鼓励开发者试用**：团队积极邀请用户通过 VS Code 扩展和`@typescript/native-preview`包体验预览版，并反馈问题以帮助完善。

### [React Server Components 中的关键安全漏洞 – React](https://react.dev/blog/2025/12/03/critical-security-vulnerability-in-react-server-components)

**原文标题**: [Critical Security Vulnerability in React Server Components – React](https://react.dev/blog/2025/12/03/critical-security-vulnerability-in-react-server-components)

React 团队披露了一个严重安全漏洞（CVE-2025-55182），影响 React Server Components，可能导致未经身份验证的远程代码执行，需立即升级相关包至修复版本。

- 🚨 **漏洞概述**：React Server Components 存在未经身份验证的远程代码执行漏洞（CVSS 10.0），攻击者可通过恶意 HTTP 请求在服务器上执行代码。
- 📅 **影响版本**：涉及 react-server-dom-webpack、react-server-dom-parcel、react-server-dom-turbopack 的 19.0、19.1.0、19.1.1、19.2.0 版本。
- ⚡ **修复版本**：立即升级至 19.0.1、19.1.2 或 19.2.1 版本；若应用未使用服务器或相关框架/打包工具则不受影响。
- 🔧 **受影响框架**：包括 Next.js、React Router、Waku、@parcel/rsc、@vitejs/plugin-rsc 和 rwsdk 等，需按官方指南升级。
- 🛡️ **临时缓解**：部分托管服务商已实施临时防护措施，但仍需尽快更新应用以彻底修复。
- 📝 **升级指南**：针对不同框架（如 Next.js 需升级至指定版本）和工具提供了具体 npm 安装命令，确保依赖包更新至最新安全版本。
- ⏳ **时间线**：漏洞于 11 月 29 日报告，12 月 1 日完成修复，12 月 3 日公开发布并分配 CVE 编号。
- 🙏 **致谢**：感谢 Lachlan Davidson 通过 Meta 漏洞赏金计划发现并协助修复此漏洞。

- 🚨 **漏洞严重性**：该漏洞被评为 CVSS 10.0 最高级别，攻击者可通过特制请求在未修复环境中执行远程代码。

- ⚡ **行动要求**：用户必须升级到对应发布线的最新修复版本；使用受影响测试版的应降级至稳定版或升级到已修复的测试版。
- 🔍 **漏洞发现**：由 Lachlan Davidson 发现并负责任地披露，为保护未升级用户，公告中未提供详细技术细节。

- 🔗 **OIDC 可信发布**：在 CI/CD 中使用 OIDC 替代长期令牌，实现无需存储密钥的安全发布流程。
- 🌳 **减少依赖树规模**：优先使用原生 JavaScript 功能，减少第三方依赖，以降低攻击面和维护风险。

TypeScript 的严格性并非单调递增：启用 `strictNullChecks` 和 `noImplicitAny` 的顺序不同，可能导致错误先出现后消失，这与“越严格错误越多”的直觉相悖。这一现象源于类型推断在特定配置下的边缘情况，虽对实际开发影响轻微，但揭示了严格模式过渡期的有趣行为。

- 🧩 **严格选项的相互作用**：`strictNullChecks` 和 `noImplicitAny` 共同影响空数组的类型推断，不同启用顺序会导致 `any[]`、`number[]` 或 `never[]` 等不同结果。
- 🔄 **非单调的错误变化**：若先启用 `strictNullChecks`，可能产生 `never[]` 类型错误；再启用 `noImplicitAny` 后错误反而消失，违反“越严格错误越多”的预期。

- 🛠️ **简单的解决方案**：可通过显式注解（如 `const array: number[] = []`）、调整启用顺序或同时启用两个选项来避免此类临时错误。
