### [Shadcn 单选按钮令人难以置信的过度复杂性](https://paulmakeswebsites.com/writing/shadcn-radio-button/)

**原文标题**: [The Incredible Overcomplexity of the Shadcn Radio Button](https://paulmakeswebsites.com/writing/shadcn-radio-button/)

作者在尝试更新网页应用中的单选按钮样式时，发现项目使用了 Shadcn 组件库中的复杂 React 组件，这引发了对现代前端开发过度复杂化的反思。通过分析代码，作者揭示了 Shadcn 依赖 Radix 底层库、引入第三方图标、使用大量 ARIA 属性模拟原生功能等现象，并指出这种实现方式违背了 ARIA 使用原则。文章强调原生 HTML 单选按钮完全可以通过简单 CSS 实现定制样式，而当前趋势却增加了不必要的依赖、代码量和性能开销。

- 🎨 **样式定制需求**：作者最初认为调整单选按钮样式很简单，因为 HTML 原生支持
- 📦 **组件库依赖**：项目使用 Shadcn 的 RadioGroup 组件，需导入 Radix 库和图标库

### [AGENTS.md 在我们的代理评估中表现优于技能 - Vercel](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals)

**原文标题**: [AGENTS.md outperforms skills in our agent evals - Vercel](https://vercel.com/blog/agents-md-outperforms-skills-in-our-agent-evals)

我们原本预期技能（skills）是教授 AI 编码助手框架特定知识的最佳方案。但在针对 Next.js 16 API 构建评估后，我们发现了意外结果：直接嵌入 AGENTS.md 的 8KB 压缩文档索引实现了 100% 通过率，而技能即使在明确指令下最高也只有 79%。没有指令时，技能的表现与完全没有文档无异。

- 🤔 **问题背景**：AI 编码助手依赖的训练数据会过时。Next.js 16 引入了`use cache`、`connection()`等新 API，但当前模型训练数据中尚未包含，导致助手生成错误代码或使用旧模式。

- 🔄 **两种方法**：我们测试了两种方案：一是**技能**（按需调用的标准化知识包），二是**AGENTS.md**（项目根目录中为助手提供持久上下文的 Markdown 文件）。

- 🎯 **初始押注技能**：技能看似是理想的抽象方案——助手在需要时调用技能获取文档，实现关注点分离和最小上下文开销。

- ⚠️ **技能触发不可靠**：评估显示，56% 的情况下技能从未被调用。默认配置下技能通过率（53%）与无文档基线完全相同，未带来任何提升。

- 💡 **显式指令有效但脆弱**：在 AGENTS.md 中添加调用技能的明确指令后，触发率升至 95% 以上，通过率提升至 79%。但指令的细微措辞变化会导致结果显著差异，方案显得脆弱。

- 🧪 **构建可信评估**：我们强化了评估套件，移除测试泄漏、解决矛盾点，并专注于测试未包含在训练数据中的 Next.js 16 API 行为。

- 🚀 **直觉验证**：我们尝试将文档索引直接嵌入 AGENTS.md，并添加关键指令：“对于任何 Next.js 任务，优先采用检索引导推理而非预训练引导推理”。

- 📊 **意外结果**：AGENTS.md 文档索引在强化评估中实现了 100% 通过率，在构建、检查和测试各项均获满分，显著优于技能方案（即使有显式指令）。

- 🔍 **原因分析**：被动上下文优于主动检索可能因为：1) **无决策点**（信息已存在，无需决定是否查找）；2) **持续可用性**（内容在每个回合的系统提示中）；3) **无顺序问题**（避免“先读文档还是先探索项目”的决策）。

- 📦 **解决上下文膨胀**：通过压缩将初始 40KB 文档注入减少至 8KB（压缩 80%），同时保持 100% 通过率。压缩格式使用管道分隔结构，将文档索引打包到最小空间。

- 🛠️ **亲自尝试**：可通过命令`npx @next/codemod@canary agents-md`一键为 Next.js 项目设置，该命令会检测 Next.js 版本、下载匹配文档并注入压缩索引到 AGENTS.md。

- 📝 **对框架作者的启示**：技能并非无用，更适合垂直、特定操作的工作流。但对于通用框架知识，被动上下文目前优于按需检索。建议：不要等待技能改进、积极压缩、使用未包含在训练数据中的 API 进行测试、设计便于检索的文档结构。

- 🎯 **最终目标**：将助手从预训练引导推理转向检索引导推理。AGENTS.md 是目前实现这一目标最可靠的方式。

- 🎨 **clip-path 的作用**：CSS 的`clip-path`属性用于定义元素的裁剪区域，仅显示区域内的内容，支持圆形、多边形等形状，但在 React Native 中需依赖第三方库。
- 🔧 **React Native 的差异**：与 Web 统一渲染引擎不同，React Native 通过协调原生 UI 组件渲染，因此需要跨平台实现`clip-path`，涉及 JavaScript、C++ 核心层及平台特定代码。
- 📱 **iOS 实现**：利用 UIKit 的`CALayer`掩码功能，在`RCTViewComponentView`中扩展`invalidateLayer`方法，通过`CAShapeLayer`创建路径实现裁剪。
- 🤖 **Android 实现**：通过重写`View`的`draw`方法，使用`Canvas`的`clipPath`函数应用裁剪，需在`BackgroundStyleApplicator`中统一处理以确保所有组件支持。
- 🚀 **架构集成**：实现需修改 React Native 的新架构（Fabric），包括更新 Shadow Node 属性、解析 CSS 值，并确保裁剪属性正确传递到原生层。
- 🔗 **未来展望**：作者已提交相关 PR，旨在将更多 CSS 功能引入 React Native，让开发者能更无缝地复用 Web 样式知识到移动应用开发中。

- 🎨 **动画控制**：利用`@starting-style`定义元素进入和退出时的起始样式，结合`transition`实现属性插值动画
- 🚫 **无 JS 依赖**：完全基于 CSS 实现，无需 JavaScript 计算位置差异，简化开发流程
- 🌐 **空间连续性**：元素从触发点自然“生长”而出，避免突兀的“瞬移”效果，增强用户体验
- ♿ **无障碍支持**：支持语义化 HTML 和`prefers-reduced-motion`媒体查询，确保可访问性
- 🔄 **灵活应用**：适用于对话框、弹出层、工具提示、全屏接管等多种交互场景，并可向不同元素变形
