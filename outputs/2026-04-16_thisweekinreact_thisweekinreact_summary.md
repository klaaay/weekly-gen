### [Node.js流：由timneutkens提出的分叉点 · Pull Request #92252 · vercel/next.js · GitHub](https://github.com/vercel/next.js/pull/92252)

**原文标题**: [Node.js streams: Fork points by timneutkens · Pull Request #92252 · vercel/next.js · GitHub](https://github.com/vercel/next.js/pull/92252)

Next.js 项目在 PR #92252 中实现了对原生 Node.js 流的支持，旨在通过使用 Node.js 原生的 `Transform`、`PassThrough` 和 `Readable` 流来优化服务器端渲染（SSR）的性能，减少因在 Web Streams 和 Node.js Streams 之间转换而产生的开销和内存复制。

- 🚀 **核心目标**：通过启用实验性标志 `useNodeStreams`，让渲染管道端到端地使用原生 Node.js 流，以消除 Web Streams 带来的缓冲、调度不匹配和额外内存拷贝等瓶颈，提升服务端渲染吞吐量。
- 🔧 **实现方式**：重新实现了所有基于 Web `TransformStream` 的转换操作为 Node.js `Transform` 流，并引入了新的 `ReplayableNodeStream` 类来替代 Web 流的 `tee()` 操作，确保在正确的异步本地存储（ALS）上下文中进行分块传递。
- 🧩 **代码分支**：在 `app-render.tsx` 中的关键渲染点根据环境变量 `process.env.__NEXT_USE_NODE_STREAMS` 进行分支，分别调用针对 Node.js 流或 Web 流优化的渲染函数（如 `renderToNodeFlightStream` 和 `renderToWebFlightStream`）。
- 📊 **测试与集成**：更新了 CI 工作流以隔离测试 Node.js 流路径，并提供了详细的性能指标对比，显示在多数场景下性能持平或略有提升，但 `node_modules` 大小有轻微增加（+1.34 MB）。
- 🔄 **后续工作**：该 PR 是基础性改动，后续有一系列相关的 PR 在此基础上为预渲染、缓存组件、错误恢复等更多路径添加了对应的“分叉点”支持。

- 📂 垂直结构建议按功能或领域（如“仪表盘”“分析”）组织代码，使团队能更清晰地拥有和维护特定模块，提升协作效率。

JavaScript Promise 无法直接取消，但可通过永不解析的 Promise 实现函数中断，避免异常捕获干扰，并依赖垃圾回收清理挂起状态。

- 🚫 Promise 没有内置取消方法，因强制取消可能导致资源状态混乱
- ⏸️ 利用永不解析的 Promise 中断 `await`，函数会静默停止执行
- 🛡️ 相比抛出异常，此方法不会被 `try/catch` 意外捕获而破坏控制流
- 🔄 生成器（Generators）虽支持精确中断，但语法不符合主流 `async/await` 习惯
- ⚙️ 通过步骤记忆化（memoization）和事件循环机制，可实现函数分段执行与恢复
- 🧹 中断后，未被引用的 Promise 和函数状态会被垃圾回收器自动清理
- 🚧 需注意避免意外引用导致的内存泄漏，确保中断模式可靠工作
