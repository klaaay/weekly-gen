React 团队推出了新的 Hook `useEffectEvent`，旨在解决 `useEffect` 中常见的闭包陈旧和无限循环等问题，通过将状态依赖从依赖数组中移除，使副作用代码更简洁、安全。

- 🔧 `useEffectEvent` 允许创建不随渲染变化的函数，避免状态依赖导致的副作用重置或循环

### [React已变，你的Hooks也该更新了 - Matt Smith](https://allthingssmitty.com/2025/12/01/react-has-changed-your-hooks-should-too/)

**原文标题**: [
    React has changed, your Hooks should too - Matt Smith
  ](https://allthingssmitty.com/2025/12/01/react-has-changed-your-hooks-should-too/)

React Hooks 的设计理念已随 Concurrent React（React 18/19）演进，强调更清晰的数据流与架构模式，以提升应用表达力与性能。

- 🚀 **避免滥用 useEffect**：仅将其用于真实副作用（如网络请求、DOM 操作），而派生状态应通过 `useMemo` 或 `useCallback` 在渲染中计算。
- 🧩 **自定义 Hook 封装逻辑**：将领域逻辑（如窗口尺寸监听）提取为独立 Hook，保持组件专注于 UI，提升可测试性与可维护性。
- 🔄 **使用 useSyncExternalStore 管理订阅状态**：适用于浏览器 API（如 `matchMedia`）或外部存储（如 Redux），确保同步更新并避免渲染不一致。
- ⚡ **利用并发特性优化交互**：通过 `useDeferredValue` 延迟派生值计算，或使用 `startTransition` 推迟状态更新，保持界面响应流畅。
- 🧪 **增强 Hook 可测试性与调试**：借助 React DevTools 直观调试，并将逻辑与 UI 分离，便于直接测试 Hook 而无需渲染组件。
- 🌐 **拥抱数据优先的 React 演进**：关注 Server Components、`use()`、服务端操作等新范式，减少对 `useEffect` 的依赖，转向更清晰的服务器/客户端数据流架构。

### [AI编码代理如何在我们应用中埋下定时炸弹 | acusti.ca](https://acusti.ca/blog/2025/12/09/how-ai-coding-agents-hid-a-timebomb-in-our-app/)

**原文标题**: [How AI Coding Agents Hid a Timebomb in Our App | acusti.ca](https://acusti.ca/blog/2025/12/09/how-ai-coding-agents-hid-a-timebomb-in-our-app/)

概述：本文讲述了开发者在构建AI网站生成器Outlyne时，因AI编码助手删除了一个关键注释，导致代码中无限递归的安全约束被破坏，最终引发应用崩溃的经过。文章详细描述了问题如何被React 19的Activity组件掩盖，以及从调试到解决的整个过程，并总结了在AI辅助编程时代，用测试而非注释来确保代码结构安全的重要性。

- 🐛 AI编码助手在重构时删除了一个关于无限递归的关键注释，导致后续代码修改移除了防止递归的安全约束（readOnly属性）
- 🔄 应用架构中，页眉页脚预览组件本应通过readOnly=true避免递归渲染编辑界面，但约束消失后形成了无限组件树
- ⏳ React 19的Activity组件使用隐藏的低优先级渲染，意外掩盖了问题，使应用在崩溃前能正常运行数分钟
- 🕵️ 调试过程异常困难，因为崩溃延迟发生且堆栈跟踪匿名，最终通过移除Activity包装才立即暴露问题
- 🧪 核心教训：在AI辅助编程时代，关键的结构性约束必须通过测试（而非注释）来强制保证，因为注释会在重构中丢失
- ⚠️ 事件暴露了当前代码审查流程的局限性，面对AI生成的大量代码变更，需要更严格的自动化约束机制
