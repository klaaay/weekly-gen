React 19.2 引入的 useEffectEvent 钩子旨在解决 Effect 中因闭包导致的过时值问题，它允许在 Effect 回调中读取最新的 props 或 state，而无需将它们加入依赖数组，从而避免不必要的 Effect 重新执行。

- 🎯 **解决过时闭包问题**：useEffectEvent 创建稳定的函数，在调用时始终读取最新值，无需依赖项触发重新运行。
- 🔄 **替代 useRef 方案**：相比传统 useRef 加同步 Effect 的繁琐模式，useEffectEvent 减少了样板代码，更简洁直观。
- ⚙️ **工作原理**：内部通过 ref 存储最新回调，返回包装函数保持稳定引用，调用时委托给最新版本。
- 📝 **使用规则**：仅限在 Effect 内调用，不应作为 props 传递，且需紧邻使用它的 Effect 声明。
- 🚫 **避免误用**：不应用于抑制依赖项 lint 警告，而是明确区分响应式（触发 Effect）与非响应式（仅读取最新值）逻辑。
- 🔧 **适用场景**：适用于订阅、计时器等需要读取最新状态但不希望触发重新连接的场景。
- 📚 **版本要求**：React 19.2 及以上稳定支持，旧版本可使用 useRef 模式实现类似效果。

- 🚫 **避免测试实现细节**：应测试用户可见的行为而非内部实现（如直接测试钩子），确保测试真正验证功能
- 📦 **专注应用逻辑**：仅测试自身代码，避免测试第三方库或框架的内部机制（如 React 的渲染逻辑）
- 🔍 **使用语义化查询**：优先使用`getByRole`、`getByLabelText`等语义化查询函数，提升测试可读性和可访问性

- 🌧️ **测试异常路径**：不仅测试“成功路径”，还需覆盖错误处理、验证失败等异常场景，确保健壮性
- 📸 **避免过度使用快照**：快照测试适用于简单输出，但复杂对象或 DOM 快照难以维护；应聚焦关键数据断言
- 📁 **良好测试结构**：使用`describe`和`it`组织测试，命名清晰，便于查找和维护相关测试用例
- ⚡ **保持测试快速运行**：利用`--watch`模式快速反馈，避免不必要的等待，提升开发效率
- 🏷️ **避免基于类名查询**：不应依赖 CSS 类名选择元素，应使用语义化查询，减少与样式的耦合
- 🎨 **避免断言具体类名**：除非测试主题相关逻辑，否则不应断言元素是否包含特定 CSS 类
- 🧠 **避免测试状态管理内部**：应测试组件行为（如界面变化），而非 Redux 等状态管理的内部动作分发
- ✅ **使用有意义的断言**：避免仅用`toBeDefined()`，应断言具体值或结构，以真正验证逻辑正确性
- 🏗️ **使用辅助函数和固件**：通过`createUser`等辅助函数生成测试数据，提升测试可读性和可维护性
- 🛠️ **创建渲染辅助函数**：对于多层 Provider 包装的组件，创建`renderWithProviders`等辅助函数简化测试设置

- 💡 **解释预期值**：在断言中使用注释或变量名说明预期值的来源，避免“魔数”难以理解
- 🔄 **避免重复实现逻辑**：测试中的预期值不应通过调用被测函数相同逻辑计算，否则无法发现逻辑错误
- 🚫 **删除无用测试**：避免测试不可能失败的内容（如静态文本），确保每个测试都有实际价值
- 🎯 **避免测试模拟本身**：模拟应简单返回预设值，而非重新实现复杂逻辑，否则测试的是模拟而非真实行为
- 🧪 **防止测试间泄漏**：每个测试应完全独立，不依赖执行顺序；使用`beforeEach`/`afterEach`清理状态
- 🔑 **避免硬编码签名哈希**：如 JWT 令牌等签名数据，应在测试中注释生成方式或使用工具动态生成
- ⚖️ **避免测试中的条件语句**：测试中不应使用 if/else，除非在`it.each`中处理不同输入场景
- 💬 **添加注释说明意图**：使用描述性变量名和注释阐明测试目的，便于后续维护和理解
- ♿ **考虑可访问性测试**：使用`getByRole`等查询有助于确保语义化标记，可结合`isInaccessible`等工具

- 👁️ **使用 toBeVisible 断言可见性**：断言元素应对用户可见时，使用`toBeVisible()`而非`toBeInTheDocument()`
- 🌐 **模拟网络请求**：使用 Mock Service Worker 或 fetch 模拟避免真实 API 调用，提升测试速度和可控性
- 🖱️ **优先使用 userEvent**：模拟用户交互时，`userEvent`比`fireEvent`更贴近真实行为，优先选用
- 🧩 **使用自定义匹配器**：通过`expect.extend`创建自定义匹配器（如`toBeValidUser`），简化复杂对象断言
- 🌍 **测试不同时区**：涉及日期时间的功能需测试多种时区场景，避免因本地时区假设导致 bug
- ⏰ **使用假定时器**：涉及定时器的测试应使用`vi.useFakeTimers`，避免真实等待，加速测试执行
- 🏷️ **模拟数据使用唯一字符串**：为模拟数据使用具描述性的唯一值（如`user-john-123`），便于调试时定位问题

### [日志记录之痛——你的日志在欺骗你](https://loggingsucks.com/)

**原文标题**: [Logging Sucks - Your Logs Are Lying To You](https://loggingsucks.com/)

本文探讨了传统日志记录在现代分布式系统中的局限性，并提出了通过“宽事件”或“规范日志行”来改进日志记录的方法。传统日志通常分散、缺乏上下文，导致调试困难；而宽事件方法强调为每个请求在每个服务中记录一个包含所有相关上下文（如用户信息、业务状态、错误详情等）的丰富结构化事件。结合尾部采样策略，可以在控制成本的同时确保关键事件（如错误、慢请求）被完整记录，从而将日志从简单的文本搜索转变为强大的分析工具，显著提升故障排查和系统监控的效率。

- 🚨 **传统日志的缺陷**：传统日志记录方式在分布式系统中显得力不从心，日志分散、缺乏上下文，导致调试时难以追踪完整请求链路和根本原因。
- 🔍 **字符串搜索的局限性**：基于文本的日志搜索无法理解数据结构或跨服务关联事件，且同一数据（如用户 ID）可能以多种格式记录，增加了排查复杂度。
- 📊 **结构化日志与宽事件的区别**：结构化日志（如 JSON 格式）是基础，但宽事件强调为每个请求记录一个包含高基数、高维度数据的完整上下文事件，而非多个零散日志行。
- 🛠️ **OpenTelemetry 的定位**：OpenTelemetry 提供了统一的遥测数据收集和导出协议，但不会自动添加业务上下文或决定记录内容，仍需开发者有意识地实施。
- 💡 **宽事件的核心优势**：通过在每个服务中为请求记录一个包含用户、业务、错误等全方位信息的“宽事件”，调试时可直接查询结构化数据，快速定位问题。
- ⚙️ **实现模式示例**：建议通过中间件在请求生命周期中构建宽事件，并在处理程序中逐步丰富业务上下文，最后在请求完成时统一发出。
- 📉 **尾部采样控制成本**：采用尾部采样策略，确保错误、慢请求、VIP 用户等关键事件 100% 保留，而对正常请求仅随机采样少量，以平衡数据完整性和存储成本。
- 🔄 **常见误解澄清**：宽事件不同于单纯的结构化日志或追踪，它强调上下文丰富性；高基数数据在现代列式数据库中可高效处理；日志和指标可基于同一数据源。
- 🎯 **改进后的效益**：宽事件将调试从“考古挖掘”转变为“数据分析”，支持复杂查询和聚合，能快速识别根因，提升运维效率和系统可靠性。

- 🚀 **功能稳定化**：`require(esm)` 已在所有支持的 Node.js LTS 版本（v20.19.0+、v22.12.0+）中取消实验标志并标记为稳定。

- ✨ **简化配置**：包维护者现在可以简化 `package.json`，移除 ESM 到 CommonJS 的转译步骤和分离的导出条件，直接声明 ESM 并指定支持的 Node.js 版本。

- 🎯 **适用场景**：信号系统更适合需要即时同步的 UI 渲染，而查询架构更适用于对时序要求较宽松的编译器工具链。
