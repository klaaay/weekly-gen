### [状态更新内部工作原理 | React 内部机制解析](https://inside-react.vercel.app/blog/how-state-updates-work-internally)

**原文标题**: [How state updates work internally | Inside React](https://inside-react.vercel.app/blog/how-state-updates-work-internally)

本文深入探讨了 React 状态更新的内部机制，解释了为什么直接调用`setState`后立即`console.log`状态值不会显示更新，以及多次调用`setState`时为何有时只更新一次，有时却能累积更新。文章通过代码示例和内部原理分析，阐述了状态更新的异步性、批处理机制以及函数式更新与直接传递值的区别。

- 🧠 React 状态更新是异步的，调用`setState`只是将更新加入队列并安排重新渲染，不会立即改变当前渲染中的状态变量。
- 🔄 在事件处理函数中，多个`setState`调用会被 React 批量处理，直到函数执行完毕后才开始渲染阶段。
- 📊 使用`setState(value)`时，如果多次基于当前状态值更新，由于状态值在渲染前未变，最终只会应用最后一次的更新值。
- ⚙️ 使用`setState(prev => prev + 1)`的函数式更新能确保每次更新都基于最新的计算状态，从而实现累积更新。
- 🌳 React 内部使用 Fiber 架构管理组件树，状态和更新队列存储在 Fiber 节点的`memoizedState`链表中。
- 🚦 更新优先级通过“车道”（lane）系统管理，确保高优先级更新（如用户交互）能及时响应。
- ⏳ `dispatchSetState`函数负责创建更新对象、加入队列，并调度渲染，但不会中断当前 JavaScript 执行。

**原文标题**: [Source Maps: Shipping Features Through Standards | Bloomberg JS Blog](https://bloomberg.github.io/js-blog/post/standardizing-source-maps/)

Source Maps 是连接开发源代码与编译后代码的关键技术，经历了从非正式协作到正式标准化的过程，如今在 ECMA-426 标准下持续演进，以支持更复杂的调试需求和开发工具。

- 🗺️ **源映射的作用**：在复杂 Web 开发中，源映射将优化、压缩后的代码映射回原始源代码，使调试成为可能。
- 📜 **历史背景**：源映射最初缺乏官方标准，依赖非正式协作（如共享文档）运行了十年，限制了新功能的添加和工具的发展。
- 🔧 **技术构成**：源映射是一个 JSON 文件，包含版本、源文件列表、映射数据等字段，其中 `mappings` 字段使用 Base64 VLQ 编码以提高效率。
- 📉 **版本演进**：Revision 3（2011 年）通过改用分段映射和相对编码，大幅减小了源映射的文件大小，成为至今使用的格式。
- 🚫 **非标准扩展**：在没有标准的情况下，社区通过 `x_` 前缀字段（如 `x_google_ignoreList`）添加了忽略列表等功能，但协调复杂功能（如函数名映射）仍困难。
- 🤝 **标准化进程**：2023 年，多家公司组成 TC39-TG4 任务组，在 Ecma International 下推动源映射标准化，最终于 2024 年发布为 ECMA-426 标准。
- 🔮 **未来特性**：新标准正在开发“作用域映射”和“范围映射”等高级功能，以提升调试精度，支持内联函数、变量重命名等复杂编译场景。
- 🌍 **社区协作**：源映射的发展体现了开源生态的协作精神，鼓励工具开发者、浏览器厂商等共同参与标准的完善与实施。

- 🛡️ **外部 JSON 数据**：使用 Zod 等模式验证库替代直接断言，确保运行时数据结构的正确性，避免因 API 变更导致的隐蔽错误。
- 🔧 **泛型边界**：在通用工具函数（如`safeFetchJson`）中集中进行一次类型断言，将验证责任转移给调用方，保持应用代码的纯净。

- 🗃️ **ORM JSON 字段**：在数据库模式定义层（如 Drizzle 的`$type`）直接声明 JSON 字段的类型，避免在每次查询时重复进行类型断言。
- 🤖 **代理与生成代码**：对于 Proxy 模式或第三方生成的代码，可接受必要的类型断言并使用行内禁用注释，将精力集中在可控制的逻辑上。

核心原则是追求零**未经检查的**类型断言，而非零断言。每个断言都应是深思熟虑后的主动选择，而非下意识的习惯。

### [介绍 view-transitions-mock：用于同文档视图过渡的非视觉 Polyfill – Bram.us](https://www.bram.us/2026/03/11/view-transitions-mock-is-a-non-visual-polyfill-for-same-document-view-transitions/)

**原文标题**: [Introducing view-transitions-mock: A non-visual Polyfill for Same-Document View Transitions – Bram.us](https://www.bram.us/2026/03/11/view-transitions-mock-is-a-non-visual-polyfill-for-same-document-view-transitions/)

view-transitions-mock 是一个非视觉化的 Polyfill，用于实现同文档视图过渡（Same-Document View Transitions）的 JavaScript API 兼容，它模拟了完整的 API 接口但不执行动画效果，从而在不支持原生视图过渡的浏览器中确保代码正常运行。

- 🛠️ **解决兼容性问题**：该工具填补了浏览器对 View Transition API 支持不全的空白，允许开发者编写无需条件判断的未来兼容代码。
- 📜 **模拟完整 API**：它实现了规范中的全部 JavaScript API 表面，包括 `document.startViewTransition()`、`ViewTransition` 类及其承诺（promises），但不涉及视觉伪树和 CSS 动画。
- ⚙️ **简单集成**：通过 npm 安装并调用 `register()` 即可启用，无需修改现有视图过渡逻辑。
- 🔧 **可配置注册**：支持通过参数调整注册行为，如忽略类型支持或强制启用模拟，便于测试和适配不同浏览器环境。
- 🧪 **演示与测试**：提供在线演示页面，展示在无原生支持浏览器中代码不崩溃的效果，并可通过强制注册观察模拟行为。
