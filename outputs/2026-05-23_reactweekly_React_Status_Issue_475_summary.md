### [结构共享、selectAtom以及为何你的jotai原子过度渲染 | Peter Piekarczyk](https://www.peterp.me/articles/jotai-structural-sharing-vs-selectatom/)

**原文标题**: [Structural sharing, selectAtom, and why your jotai atoms re-render too much | Peter Piekarczyk](https://www.peterp.me/articles/jotai-structural-sharing-vs-selectatom/)

### 概述总结
jotai 原子值通过 `Object.is` 比较决定是否触发重渲染。当派生原子每次构造新对象时，会导致不必要的重渲染。解决方案是优先分解为原始值原子，其次使用 `selectAtom` 或结构共享。

- 📊 **核心问题**：jotai 使用 `Object.is` 比较原子值，新对象引用总是不同，导致所有订阅者重渲染
- 🔧 **`selectAtom` 适用场景**：处理 `.filter/.map` 产生的新数组，或需要保持对象标识稳定的聚合结果
- 🧩 **优先分解策略**：将复合对象拆分为多个原始值原子（如 `countAtom`、`totalAtom`），利用 `Object.is` 自动去重
- 🚀 **结构共享方案**：通过 `replaceEqualDeep` 保留未变化子树的旧引用，适用于列表等动态结构
- ⚡ **jotai-tanstack-query 优化**：避免直接使用 `useAtomValue(rawQueryAtom)`，应派生具体字段原子，可将 44 倍重渲染降为 1 倍
- 🏗️ **列表处理三种方案**：无共享（简单但重渲染多）、结构共享（需写时遍历）、归一化（`atomFamily` 精确更新）
- ❓ **jotai 未内置原因**：保持最小化设计，避免对复杂类型（类实例、Date 等）的默认处理，且最佳方案取决于数据层

**原文标题**: [Release v21.0.0 · facebook/relay · GitHub](https://github.com/facebook/relay/releases/tag/v21.0.0)

Relay v21.0.0 正式发布，带来多项重大更新，包括对 TypeScript 的一流支持、现代化的 Flow 类型生成、新的错误处理能力以及实验性的 React Server Components 支持。此版本包含多项破坏性变更，升级前请仔细阅读迁移说明。

- 🎉 **TypeScript 一流支持**：`relay-runtime` 和 `react-relay` 现在直接包含 `.d.ts` 类型定义，不再需要 `@types/relay-runtime`，并且 Node.js ESM 命名导入现在可以正常工作。
- 🔄 **Flow 用户需采取行动**：生成的类型现在使用现代 Flow 语法，例如 `(expr as Type)` 替换 `(expr: Type)`，`$ReadOnlyArray` 变为 `ReadonlyArray` 等。升级后必须重新运行 Relay 编译器以重新生成所有产物。
- ❌ **`@live_query` 指令已移除**：请迁移到 `@client_polling(interval: ...)` 实现轮询，或使用 `@live` 实现服务器推送更新。
- ✍️ **Relay Resolvers 新语法**：`enable_legacy_verbose_resolver_syntax` 标志已移除，旧的 docblock 语法不再有效。新的 `@relayField` 和 `@relayType` 标签已取代通用的 `@RelayResolver` 标签。编译器可通过 `relay-compiler codemod fix-all` 命令自动迁移。
- ⚖️ **`@oneOf` 输入类型更严格**：现在生成更严格的判别联合类型，一次只允许一个属性，从而在编译时捕获同时设置多个字段的错误。
- 🆕 **新功能**：`@catch` 现在支持 `@connection` 字段和客户端边字段；`@alias` 指令现在可用于片段定义；新增实验性编译器 CLI 工具；以及实验性的 React Server Components API (`react-relay/rsc_EXPERIMENTAL`)。
- 🐛 **重要 Bug 修复**：修复了大型模式（1000+ 字段）打印时的栈溢出问题；`@raw_response_type` 中抽象类型的选择现在在具体类型可能未实现抽象类型时被正确标记为可选；编译器现在能正确验证并报告跨片段的冲突 `@match` 字段；`--watch` 模式的稳定性得到显著提升。
- 🔧 **破坏性变更**：移除了 `@live_query` 指令支持；`@oneOf` 输入对象类型生成更严格；移除了 `enable_legacy_verbose_resolver_syntax` 标志；`mixed` 变为 `unknown`，`$NonMaybeType` 变为 `NonNullable` 等。
- 📚 **文档改进**：更新了 Relay Resolver 文档以使用新的 `@relayType`/`@relayField` 语法；修复了多处拼写和语法错误；新增了术语表定义；重写了分页文档；并创建了“最佳实践”部分。

### [GitHub - mockingbot/react-native-zip-archive: 用于React Native的ZIP归档工具 · GitHub](https://github.com/mockingbot/react-native-zip-archive)

**原文标题**: [GitHub - mockingbot/react-native-zip-archive: Zip archive utility for react-native · GitHub](https://github.com/mockingbot/react-native-zip-archive)

React Native Zip Archive 是一个用于 React Native 的压缩/解压缩工具库，支持 ZIP 文件的创建、解压、密码保护、进度监听等功能，兼容 iOS 和 Android 平台。

- 📦 **核心功能**：支持文件夹或文件数组的压缩（`zip`）、解压（`unzip`）、带密码的压缩/解压（`zipWithPassword`/`unzipWithPassword`），以及从 Android assets 解压（`unzipAssets`）。
- 🔐 **加密支持**：提供 STANDARD、AES-128、AES-256 三种加密类型，但 iOS 对文件数组仅支持 STANDARD 加密。
- 📊 **进度监听**：通过 `subscribe` 订阅全局进度事件，可获取 0-1 的进度值和文件路径，适合显示进度条。
- 📏 **文件信息**：`getUncompressedSize` 可获取 ZIP 内所有文件的未压缩总大小（字节），Android 支持自定义字符集。
- 🛠️ **平台差异**：压缩级别在 iOS 仅对文件夹有效；字符集参数仅 Android 支持；`unzipAssets` 为 Android 独占功能。
- 📱 **版本要求**：需 React Native ≥ 0.70、iOS ≥ 15.5、Android API ≥ 23，v8.0 需启用新架构。
- 🚀 **安装与使用**：通过 npm 安装，iOS 需执行 `pod install`，提供清晰的 API 示例和常量（如 `DEFAULT_COMPRESSION`）。
- 🧪 **测试与示例**：包含 Expo 和纯 RN 两个 playground 应用，覆盖所有 API 方法，并集成 Maestro E2E 测试。
- 📄 **许可证与资源**：MIT 许可证，GitHub 上有 472 星、166 分支，49 个版本发布，主要语言为 TypeScript。
