### [在Next.js 16中实现next-intl国际化与'use cache'功能 | Aurora Scharff](https://aurorascharff.no/posts/implementing-nextjs-16-use-cache-with-next-intl-internationalization/)

- 🚫 **兼容性问题**：Next.js 16的'use cache'与next-intl的getTranslations()不兼容，因为后者默认依赖headers()读取请求头信息
- 🌐 **问题根源**：next-intl通过中间件将locale作为请求头传递，避免手动传递locale参数，但这会导致动态渲染
- 🔧 **临时解决方案**：
  - 配置generateStaticParams实现静态渲染
  - 在布局和页面中调用setRequestLocale
  - 需要缓存的组件需显式接收locale参数
- ⚡ **缓存组件使用**：通过传递locale参数给getTranslations()，避免读取headers()，使'use cache'正常工作
- 🎯 **未来方案**：Next.js将推出next/root-params API，让组件能直接访问路由参数，彻底解决兼容性问题
- 📝 **代码模式**：使用Suspense边界处理动态组件，缓存组件包含在静态外壳中实现部分预渲染

**原文标题**: [CVE-2025-11953 Critical RCE in React Native CLI](https://jfrog.com/blog/CVE-2025-11953-critical-react-native-community-cli-vulnerability/)

JFrog安全团队发现React Native开发工具包存在严重远程代码执行漏洞CVE-2025-11953，该漏洞允许攻击者通过开发服务器执行任意系统命令，影响范围覆盖全球约200万周下载量。

- 🚨 **漏洞等级**：CVSS 9.8分关键级漏洞，影响@react-native-community/cli-server-api 4.8.0至20.0.0-alpha.2版本
- 🌐 **攻击场景**：结合React Native核心代码的网络安全问题，开发服务器会默认暴露到外部网络，允许未授权攻击者远程触发漏洞
- ⚙️ **影响平台**：Windows系统可实现完整命令执行，macOS/Linux系统存在受限代码执行风险
- 📱 **受影响群体**：使用Metro开发服务器且运行npm start/react-native start等命令的React Native开发者
- 🛡️ **修复方案**：立即升级cli-server-api至20.0.0+版本，或启动时添加--host 127.0.0.1参数限制本地访问
- 🔍 **技术原理**：开发服务器的/open-url端点未过滤用户输入，直接传递给open()函数执行系统命令
- ⚡ **响应状态**：Meta安全团队已及时修复漏洞，建议开发者立即检查项目依赖并采取防护措施

- 🔗 next-rspack 集成：在 Next.js 16 中集成自定义 Rust 绑定，构建性能提升 24%，开发性能提升 10%。
